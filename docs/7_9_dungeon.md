좋아. 아래는 네가 정한 간소화 원칙을 반영한 **#17 던전 시스템 기획서** 초안이야.
기준은 **“복잡한 패턴 공략 콘텐츠가 아니라, 메인 성장 루프를 보조하는 일일 성장 공급처”**로 잡았어.

---

# #17 던전 시스템 기획서

## 1. 시스템 개요

### 1-1. 목적

던전은 메인 지역 진행만으로 부족해지기 쉬운 성장 자원을 안정적으로 공급하는 콘텐츠다.
유저는 **던전 티켓**을 사용해 필요한 자원을 선택적으로 획득하고, 이를 통해 다시 성장/장비/스킬 강화로 돌아가게 된다.

### 1-2. 설계 의도

이 게임의 던전은 복잡한 기믹형 콘텐츠가 아니라, 아래 목적에 집중한다.

* 메인 진행 정체 구간 해소
* 일일 접속 동기 제공
* 부족한 성장 재화 선택 파밍
* 시즌패스 / 퀘스트 / 이벤트와 연결되는 반복 플레이 지점 제공

### 1-3. 콘텐츠 범위

간소화 원칙에 따라 던전은 3종만 운영한다.

1. **골드 던전**
   성장/장비/스킬 강화에 필요한 골드 수급처

2. **장비 던전**
   장비 소환권 수급처

3. **스킬 던전**
   스킬 소환권 수급처

### 1-4. 핵심 특징

* 공통 입장 재화: **던전 티켓**
* 3종 던전 모두 **상시 개방**
* 던전별로 **스테이지형 난이도**
* 1회 클리어 후 **소탕 가능**
* 전투는 자동 전투 기반
* 장비/스킬 프리셋은 **#21 스킬 세팅** 기준으로 적용
* 직접 장비 드랍 없이 **소환권 중심 보상**
* 다이아는 **첫 클리어 보상** 위주 지급

## 2. 규칙 설명

### 2-1. 해금 규칙

* 던전 기능은 **초반 메인 진행 후 해금**
* 권장안: **지역 3 진입 시 오픈**
* 오픈 후 3종 던전이 모두 노출되며, 각 던전은 1스테이지부터 도전 가능

### 2-2. 입장 규칙

* 던전 1회 입장 시 **던전 티켓 1장** 소모
* 티켓은 모든 던전에서 공용으로 사용
* 티켓은 매일 초기화 시 일정 수량 지급
* 권장안:

  * 일일 기본 지급: **10장**
  * 최대 보유량: **20장**
* 추가 티켓은 상점/이벤트/우편을 통해 지급 가능

### 2-3. 진행 규칙

* 각 던전은 **스테이지 방식**으로 구성
* 이전 스테이지를 클리어해야 다음 스테이지 해금
* 유저는 각 던전에서 자신이 클리어한 최고 단계까지 재도전 가능
* 도전 시 현재 장착 중인:

  * 성장 수치
  * 장비 상태
  * 스킬 장착 프리셋(#21)
    를 기준으로 전투가 진행된다

### 2-4. 전투 규칙

* 전투는 **자동 전투**
* 조작형 스킬 입력 없음
* 전투 시작 시 서버가 유저의 전투 스냅샷을 고정
* 클리어 조건:

  * 제한 시간 내 마지막 적 또는 보스 처치
* 실패 조건:

  * 제한 시간 초과
  * 캐릭터 사망
* 권장 제한 시간: **60초**

### 2-5. 보상 규칙

각 던전은 1개의 핵심 보상을 중심으로 설계한다.

| 던전 종류 | 핵심 보상  | 보조 보상           | 비고          |
| ----- | ------ | --------------- | ----------- |
| 골드 던전 | 골드     | 없음 또는 소량의 추가 골드 | 성장 재화 공급    |
| 장비 던전 | 장비 소환권 | 소량 골드           | 직접 장비 드랍 없음 |
| 스킬 던전 | 스킬 소환권 | 소량 골드           | 직접 스킬 드랍 없음 |

추가 규칙:

* 각 스테이지 **첫 클리어 시 다이아 1회 지급**
* 첫 클리어 보상은 계정 기준 1회만 지급
* 길드 코인은 던전에서 지급하지 않음
* 던전 보상은 시즌패스/퀘스트 조건 카운트에 반영 가능

### 2-6. 소탕 규칙

* 해당 스테이지를 **최초 1회 이상 클리어**하면 소탕 가능
* 소탕 시에도 던전 티켓 1장 소모
* 소탕 결과는 즉시 지급
* 실패 개념 없이 확정 지급
* 소탕 가능 범위는 **해당 던전 최고 클리어 스테이지 이하**

### 2-7. 밸런스 규칙

던전은 메인 진행보다 약간 낮은 부담으로 반복할 수 있어야 한다.

권장 기준:

* 현재 유저가 막 도달한 최고 스테이지는 **실패 가능성 존재**
* 직전 클리어 스테이지는 **안정적으로 반복 가능**
* 최신 해금 단계 기준 목표 클리어율: **30~50%**
* 1단계 낮은 스테이지 기준 목표 클리어율: **70% 이상**

### 2-8. 리셋 규칙

* 티켓 수급/구매 횟수 등 일일 자원은 매일 초기화
* 첫 클리어 보상은 초기화되지 않음
* 던전 최고 클리어 기록은 영구 유지
* 던전 자체는 상시 개방이므로 요일 제한 없음

## 4. UI 구성

### 4-1. 진입 위치

* 메인 화면 하단 또는 중단 메뉴에 **던전 버튼**
* 알림 조건이 있으면 레드닷 표시

  * 사용 가능한 티켓 보유
  * 클리어 가능한 신규 단계 존재
  * 퀘스트/패스 미션 연동 보상 가능

### 4-2. 던전 로비 화면

상단:

* 제목: 던전
* 보유 던전 티켓 수량
* 일일 초기화까지 남은 시간
* 도움말 버튼

중앙:

* 던전 탭 3개

  * 골드 던전
  * 장비 던전
  * 스킬 던전

하단 또는 우측:

* 선택된 던전 설명
* 현재 최고 클리어 단계
* 주요 보상 미리보기

### 4-3. 스테이지 선택 영역

선택한 던전 안에서 스테이지 리스트 또는 카드 형태로 노출

표시 정보:

* 스테이지 번호
* 권장 전투력
* 첫 클리어 여부
* 1회 보상량
* 첫 클리어 다이아 보상
* 소탕 가능 여부
* 최고 기록 시간

### 4-4. 도전 팝업

입장 직전 팝업에 다음 정보 표시

* 선택 던전 / 스테이지
* 권장 전투력
* 현재 전투력
* 사용 프리셋 정보
* 예상 보상
* 입장 시 소모 티켓 수량
* 버튼:

  * 도전
  * 소탕
  * 프리셋 변경(#21로 이동)

### 4-5. 전투 화면

간소화 버전 기준:

* 배경
* 몬스터 웨이브
* 타이머
* 유저 HP
* 보스 HP
* 스킬 발동 연출
* 배속 / 스킵 버튼

전투 자체가 핵심이 아니므로 UI는 단순하게 유지한다.

### 4-6. 결과 팝업

클리어 시:

* 클리어 문구
* 획득 보상
* 첫 클리어 추가 보상 표시
* 다음 스테이지 해금 여부
* 버튼:

  * 다음 단계
  * 같은 단계 반복
  * 나가기

실패 시:

* 실패 문구
* 실패 원인(시간 초과 / 사망)
* 권장 액션:

  * 성장으로 이동
  * 장비/스킬 강화로 이동
  * 나가기

## 5. 예외 처리

### 5-1. 티켓 부족

상황:

* 입장 또는 소탕 시 티켓 부족

처리:

* 입장 불가
* 팝업 노출: “던전 티켓이 부족합니다.”
* 상점 또는 획득처 안내 버튼 제공

### 5-2. 스테이지 미해금

상황:

* 이전 단계 미클리어 상태에서 상위 단계 선택

처리:

* 선택 불가 또는 비활성화
* 안내 문구: “이전 단계를 먼저 클리어해야 합니다.”

### 5-3. 소탕 조건 미충족

상황:

* 해당 스테이지 첫 클리어 전인데 소탕 시도

처리:

* 소탕 버튼 비활성화
* 안내 문구: “해당 단계를 1회 클리어하면 소탕이 해금됩니다.”

### 5-4. 네트워크 끊김 / 앱 종료

상황:

* 입장 후 결과 수신 전에 종료

처리:

* 서버는 run_id 기준으로 결과를 최종 확정
* 재접속 시 결과 팝업 또는 우편으로 보상 지급
* 중복 지급 방지 위해 run_id 단위 보상 상태 저장

### 5-5. 중복 요청

상황:

* 유저가 도전 버튼을 연타하거나 패킷 재전송 발생

처리:

* 클라이언트는 버튼 잠금
* 서버는 동일 요청을 idempotent 처리
* 동일 run_id 중복 생성 금지

### 5-6. 일일 초기화 시점 충돌

상황:

* 유저가 던전 화면을 열어둔 상태에서 자정 초기화 발생

처리:

* 서버 기준 값이 최종값
* 입장 요청 시 티켓 수량/초기화 상태 재검증
* 클라이언트는 최신 값으로 갱신

### 5-7. 밸런스 데이터 변경

상황:

* 운영 중 스테이지 보상/권장 전투력 변경

처리:

* 신규 입장부터 변경값 적용
* 이미 진행 중인 run은 입장 시점 스냅샷 기준 유지
* 클라이언트 버전 미스매치 시 재동기화

### 5-8. 비정상 클리어 시간

상황:

* 이론상 불가능한 짧은 시간, 비정상 보상 획득

처리:

* 서버 판정 우선
* 이상 로그 적재
* 필요 시 자동 제재/GM 검토 대상 등록

## 6. 서버/클라이언트 처리 구분

### 6-1. 서버 처리

서버 권한으로 처리해야 하는 항목

* 던전 해금 여부 검증
* 티켓 보유량 검증 및 차감
* 입장 가능 여부 판정
* 유저 전투 스냅샷 생성
* 던전 전투 결과 계산
* 보상 산출 및 지급
* 첫 클리어 여부 갱신
* 최고 클리어 단계 갱신
* 퀘스트 / 시즌패스 카운트 반영
* 로그 저장
* 중복 요청 방지
* 재접속 복구 처리

### 6-2. 클라이언트 처리

클라이언트는 표현과 조작 중심

* 던전 로비/스테이지 UI 표시
* 권장 전투력, 보상 미리보기 표시
* 프리셋 변경 UI 연결
* 전투 연출 재생
* 결과 팝업 노출
* 로컬 카운트다운 표시
* 버튼 상태 제어
* 서버 응답 수신 후 화면 갱신

### 6-3. 전투 판정 기준

권장 구조는 **서버 판정형**이다.

이유:

* 던전은 보상 가치가 높음
* 티켓 소모형 콘텐츠라 조작/변조에 취약하면 안 됨
* 소탕, 첫 클리어, 퀘스트 반영까지 모두 서버 일관성이 필요함

즉,
**클라이언트는 전투를 보여주고, 실제 결과는 서버가 확정**하는 방식이 적절하다.

### 7-3. 운영 지표

운영에서 봐야 할 핵심 지표

1. **던전 진입률**

* 일일 접속자 중 몇 %가 던전에 진입하는지

2. **티켓 소모율**

* 지급된 티켓 대비 실제 사용 비율
* 너무 낮으면 콘텐츠 매력이 부족한 것

3. **던전별 선택 비중**

* 골드/장비/스킬 중 어느 던전에 수요가 몰리는지
* 특정 보상이 과도하게 부족한지 판단 가능

4. **스테이지별 클리어율**

* 특정 구간 난이도 급상승 탐지

5. **실패 사유 비율**

* 시간 초과가 많은지
* 생존 실패가 많은지

6. **소탕 전환율**

* 최초 클리어 이후 반복 플레이가 소탕 중심으로 잘 전환되는지

7. **첫 클리어 다이아 지급량**

* 경제 밸런스 체크용

### 7-4. 운영 포인트

* 장비 던전/스킬 던전의 효율 차이가 너무 크면 특정 성장 축만 강제됨
* 골드 부족이 과도하면 전체 성장 체감이 급격히 나빠짐
* 티켓 공급이 너무 많으면 루틴 가치가 약해지고, 너무 적으면 불쾌감이 커짐
* 이벤트 기간에는 아래 운영이 가능

  * 던전 티켓 추가 지급
  * 특정 던전 2배 보상
  * 주말 한정 전체 보상 증가

### 7-5. GM/운영 도구 권장 기능

* 특정 유저 던전 기록 조회
* run_id 기준 보상 지급 이력 조회
* 티켓/보상 복구
* 스테이지 보상 배율 조정
* 던전 임시 오픈/클로즈
* 이상 플레이 유저 추출

# 정리

이 던전 설계의 핵심은 아래 4가지야.

* **복잡한 재료 대신 기존 6재화만 사용**
* **직접 드랍 대신 소환권 중심으로 단순화**
* **3종 던전만 두어 선택 구조는 유지**
* **서버 판정 + 소탕 중심으로 운영/치트 대응을 쉽게 설계**

원하면 다음 단계로 이어서
**“던전 상세 수치표(스테이지 1~20 보상/권장 전투력 예시)”**까지 바로 작성해줄게.
