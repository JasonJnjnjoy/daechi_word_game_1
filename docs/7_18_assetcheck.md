좋아. 아래는 네가 정한 공통 설계 기준을 바탕으로 정리한 **‘재화 확인’ 시스템 기획서**야.
목표는 **유저가 현재 보유 재화를 빠르게 확인하고, 부족한 재화를 어디서 얻는지 직관적으로 이해하게 하는 것**이야.

---

# 재화 확인 시스템 기획서

## 1. 시스템 개요

### 1-1. 시스템 목적

재화 확인 시스템은 유저가 현재 보유 중인 주요 재화를 한눈에 확인하고, 각 재화의 용도와 획득처를 쉽게 파악할 수 있도록 돕는 기능이다.
게임 내 성장, 소환, 던전, 길드 등 여러 시스템에서 재화가 사용되므로, 재화 정보는 항상 명확하고 일관되게 노출되어야 한다.

### 1-2. 관리 대상 재화

본 프로젝트에서는 아래 6종의 재화만 사용한다.

* 골드
* 다이아
* 장비 소환권
* 스킬 소환권
* 던전 티켓
* 길드 코인

### 1-3. 시스템 범위

재화 확인 시스템은 다음 범위를 담당한다.

* 현재 보유 재화 노출
* 재화 상세 정보 제공
* 재화 부족 시 획득처 안내
* 재화 증감 시 즉시 반영
* 재화 관련 알림 및 갱신 처리

### 1-4. 비범위

아래 항목은 본 시스템의 직접 범위에 포함하지 않는다.

* 재화 획득/소모의 실제 계산 로직
* 상점 판매 정책
* 결제 시스템 상세 처리
* 이벤트/던전/길드의 개별 보상 설계

즉, **재화 확인 시스템은 “보여주고 안내하는 역할”**에 집중한다.

## 2. 규칙 설명

### 2-1. 기본 노출 규칙

* 메인 화면 상단 HUD에는 핵심 재화만 노출한다.
* 기본 노출 재화:

  * 골드
  * 다이아
  * 장비 소환권
  * 스킬 소환권
* 나머지 재화:

  * 던전 티켓
  * 길드 코인
* 비핵심 재화는 확장 팝업 또는 재화 상세 화면에서 확인한다.

### 2-2. 노출 우선순위

상시 노출 우선순위는 다음과 같다.

1. 골드
2. 다이아
3. 장비 소환권
4. 스킬 소환권

이유:

* 사용 빈도가 높음
* 성장 및 소환과 직접 연결됨
* 유저가 가장 자주 확인하는 재화임

### 2-3. 수량 표기 규칙

* 1,000 이상: 천 단위 콤마 표기
* 큰 수량은 축약 표기 사용 가능

  * 예: 12,300 → 12.3K
  * 1,250,000 → 1.25M
* 축약 표기는 HUD에서만 사용하고, 상세 팝업에서는 실제 수치를 그대로 표시한다.

### 2-4. 재화 상세 정보 규칙

재화를 터치/클릭하면 상세 정보 팝업을 연다. 팝업에는 아래 정보를 제공한다.

* 재화 이름
* 현재 보유량
* 사용처
* 주요 획득처
* 최대 보유 제한 여부
* 충전형 재화 여부

### 2-5. 부족 재화 안내 규칙

유저가 특정 행동을 시도했을 때 재화가 부족하면 다음 규칙으로 처리한다.

* 부족 메시지 출력
* 부족한 재화 종류 명시
* 획득 가능한 경로 버튼 제공
* 이동 가능한 시스템이 있으면 바로가기 제공

예시:

* 골드 부족 → 성장 / 오프라인 보상 / 던전 안내
* 다이아 부족 → 상점 / 이벤트 안내
* 장비 소환권 부족 → 상점 / 이벤트 / 우편 안내
* 스킬 소환권 부족 → 상점 / 이벤트 / 우편 안내
* 던전 티켓 부족 → 시간 회복 여부 표시 / 다이아 구매 여부 표시
* 길드 코인 부족 → 길드 콘텐츠 참여 안내

### 2-6. 충전형 재화 규칙

던전 티켓은 충전형 재화로 처리한다.

* 최대 보유량 존재
* 일정 시간마다 자동 회복 가능
* 최대치 도달 시 추가 자연 회복 중지
* 회복 타이머는 재화 상세 팝업에서 확인 가능

### 2-7. 무료/유료 성격 구분

운영 및 밸런스 기준상 재화를 아래처럼 분류한다.

* 일반 성장 재화: 골드
* 프리미엄 재화: 다이아
* 콘텐츠 소비형 재화: 장비 소환권, 스킬 소환권, 던전 티켓, 길드 코인

이 구분은 UI 아이콘/색상/툴팁에도 일관되게 반영한다.

## 3. 데이터 구조

## 3-1. 마스터 데이터: CurrencyMaster

각 재화의 정의 정보를 저장한다.

| 필드명                   | 타입       | 설명                                 |
| --------------------- | -------- | ---------------------------------- |
| currency_id           | int      | 재화 고유 ID                           |
| currency_key          | string   | GOLD, DIAMOND, EQUIP_TICKET 등 내부 키 |
| name_kr               | string   | 재화명                                |
| desc                  | string   | 재화 설명                              |
| icon_path             | string   | 아이콘 리소스 경로                         |
| is_visible_hud        | bool     | HUD 기본 노출 여부                       |
| is_rechargeable       | bool     | 자연 회복형 여부                          |
| max_amount            | bigint   | 최대 보유량, 없으면 null                   |
| recovery_interval_sec | int      | 회복 주기, 없으면 0                       |
| recovery_amount       | int      | 회복량                                |
| acquisition_desc      | string[] | 주요 획득처 목록                          |
| usage_desc            | string[] | 주요 사용처 목록                          |
| sort_order            | int      | 표시 순서                              |

### 예시

* GOLD
* DIAMOND
* EQUIP_SUMMON_TICKET
* SKILL_SUMMON_TICKET
* DUNGEON_TICKET
* GUILD_COIN

---

## 3-2. 유저 보유 데이터: UserCurrency

유저별 보유 재화를 저장한다.

| 필드명         | 타입       | 설명       |
| ----------- | -------- | -------- |
| user_id     | bigint   | 유저 ID    |
| currency_id | int      | 재화 ID    |
| amount      | bigint   | 현재 보유량   |
| updated_at  | datetime | 최종 갱신 시각 |

복합 키:

* `(user_id, currency_id)`

---

## 3-3. 충전형 재화 데이터: UserRechargeCurrency

충전형 재화 전용 상태값을 관리한다.

| 필드명              | 타입       | 설명           |
| ---------------- | -------- | ------------ |
| user_id          | bigint   | 유저 ID        |
| currency_id      | int      | 재화 ID        |
| current_amount   | int      | 현재 수량        |
| max_amount       | int      | 최대 수량        |
| last_recovery_at | datetime | 마지막 회복 기준 시각 |

대상:

* 던전 티켓

---

## 3-4. 재화 변동 로그: CurrencyTransactionLog

재화 증감 이력을 기록한다.

| 필드명           | 타입       | 설명                           |
| ------------- | -------- | ---------------------------- |
| log_id        | bigint   | 로그 ID                        |
| user_id       | bigint   | 유저 ID                        |
| currency_id   | int      | 재화 ID                        |
| change_type   | enum     | 획득 / 소모 / 회복 / 차감 취소 등       |
| delta_amount  | bigint   | 증감 수량 (+/-)                  |
| before_amount | bigint   | 변경 전 수량                      |
| after_amount  | bigint   | 변경 후 수량                      |
| source_type   | enum     | 오프라인보상 / 던전 / 상점 / 우편 / 길드 등 |
| source_id     | bigint   | 관련 콘텐츠 ID                    |
| reason_text   | string   | 운영 확인용 설명                    |
| created_at    | datetime | 발생 시각                        |

## 4. UI 구성

## 4-1. 메인 HUD 영역

메인 화면 상단에 재화 바를 배치한다.

### 구성 요소

* 골드
* 다이아
* 장비 소환권
* 스킬 소환권
* 더보기 버튼

### 기능

* 실시간 수량 표시
* 클릭 시 상세 팝업 진입
* 수량 변동 시 간단한 증감 연출 표시

  * 예: `+500`, `-1`

### 설계 의도

상단 HUD는 전투/지역/성장 화면 어디서든 동일한 위치와 규칙으로 노출되어야 한다.
유저가 매번 재화를 찾지 않도록 하는 것이 목적이다.

---

## 4-2. 재화 전체 보기 팝업

더보기 버튼 또는 재화 바 클릭 시 전체 재화 팝업을 연다.

### 표시 항목

* 6종 재화 전체 목록
* 각 재화의 현재 수량
* 재화 설명
* 주요 획득처 아이콘 또는 텍스트
* 필요 시 이동 버튼

### 추천 UI 구조

* 상단: “보유 재화” 타이틀
* 중앙: 재화 리스트 2열 또는 1열 카드형
* 하단: 닫기 버튼

---

## 4-3. 재화 상세 팝업

특정 재화를 선택하면 상세 팝업을 띄운다.

### 표시 정보

* 재화 아이콘
* 재화 이름
* 현재 보유량
* 재화 설명
* 사용처
* 획득처
* 바로가기 버튼

### 예시

**골드**

* 설명: 성장과 장비 강화에 사용되는 기본 재화
* 사용처: 성장, 일부 시스템 강화
* 획득처: 지역 사냥, 오프라인 보상, 던전

**던전 티켓**

* 설명: 던전 입장에 필요한 티켓
* 사용처: 일일 던전 입장
* 획득처: 시간 회복, 이벤트, 상점
* 추가 표기: `다음 회복까지 12:35`

---

## 4-4. 부족 재화 팝업

재화 부족 시 전용 팝업을 노출한다.

### 표시 정보

* 부족 재화명
* 현재 보유량 / 필요량
* 획득 추천 경로
* 이동 버튼
* 닫기 버튼

### 예시 문구

* 골드가 부족합니다.
* 던전 티켓이 부족합니다.
* 장비 소환권이 부족합니다.

### 이동 버튼 예시

* 지역으로 이동
* 던전으로 이동
* 상점으로 이동
* 길드로 이동

---

## 4-5. 연출 규칙

* 재화 획득 시 해당 재화 영역에 강조 애니메이션
* 대량 획득 시 획득 출처 요약 표기 가능
* 보상 수령 직후 상단 HUD 즉시 갱신
* 수량이 최대치인 충전형 재화는 “MAX” 표시 가능

## 5. 예외 처리

## 5-1. 서버 값과 클라이언트 값 불일치

문제:

* 클라이언트 캐시 수치와 서버 실제 수치가 다를 수 있음

처리:

* 서버 값을 최종 권한 값으로 사용
* 중요 소비 직전 서버 검증 수행
* 차이가 발생하면 HUD 즉시 재동기화

---

## 5-2. 재화 부족 상태에서 연속 입력

문제:

* 유저가 버튼을 여러 번 눌러 중복 요청 발생 가능

처리:

* 첫 요청 전송 후 버튼 잠금
* 응답 전까지 중복 입력 차단
* 실패 시만 다시 활성화

---

## 5-3. 음수 재화 발생

문제:

* 잘못된 차감 로직으로 보유량이 0 미만이 될 수 있음

처리:

* 서버에서 차감 전 항상 보유량 검증
* 0 미만 결과는 저장 금지
* 비정상 시 에러 코드 반환 및 로그 적재

---

## 5-4. 충전형 재화 회복 오차

문제:

* 접속 중/오프라인/재접속 시점에 회복 수량 차이가 날 수 있음

처리:

* 회복 계산은 서버 기준 시간으로 수행
* 클라이언트는 남은 시간만 보조 표시
* 로그인 시 서버에서 재계산 후 반영

---

## 5-5. 표시 불가 수치

문제:

* 매우 큰 수치나 포맷 오류 발생 가능

처리:

* HUD에서는 최대 표기 자릿수 초과 시 축약 표기 사용
* 비정상 값은 `0`으로 대체 표시하지 않고, 에러 처리 후 마지막 정상 값 유지
* UI 깨짐 방지를 위해 문자열 길이 제한 적용

---

## 5-6. 획득처 미설정 재화

문제:

* 신규 재화 마스터 데이터 누락 시 상세 팝업 품질 저하

처리:

* 획득처/사용처 마스터값 누락 시 기본 문구 출력

  * “획득처 정보 준비 중”
* 운영 알림 로그 생성

---

## 6. 서버/클라이언트 처리 구분

## 6-1. 서버 처리

서버는 재화 수치에 대한 최종 권한을 가진다.

### 서버 담당

* 재화 보유량 저장
* 재화 획득 및 소모 처리
* 충전형 재화 회복 계산
* 보유량 검증
* 재화 부족 판정
* 재화 로그 기록
* 비정상 패턴 탐지용 데이터 제공

### 서버 원칙

* 모든 재화 증감은 서버에서만 확정
* 클라이언트 요청은 “행동 요청”만 보내고, 결과 수치는 서버가 응답

---

## 6-2. 클라이언트 처리

클라이언트는 표시와 사용자 경험을 담당한다.

### 클라이언트 담당

* HUD 표시
* 재화 팝업 표시
* 부족 안내 팝업 표시
* 수량 변동 연출
* 상세 획득처/사용처 표시
* 서버 응답 반영

### 클라이언트 원칙

* 재화 계산 확정값을 자체 결정하지 않음
* 임시 표시 가능하지만, 반드시 서버 응답으로 최종 갱신

---

## 6-3. 통신 예시

### 1) 로그인 시

* 서버 → 유저 재화 전체 정보 전달
* 클라이언트 → HUD 및 상세 팝업 초기화

### 2) 보상 수령 시

* 클라이언트 → 보상 수령 요청
* 서버 → 재화 반영 후 최신 수치 반환
* 클라이언트 → HUD 갱신 및 연출 재생

### 3) 재화 소모 행동 시

* 클라이언트 → 소비 요청
* 서버 → 보유량 검증

  * 가능: 차감 후 성공 응답
  * 불가: 부족 에러 코드 반환
* 클라이언트 → 성공 반영 또는 부족 팝업 노출

## 7. 로그 및 운영 포인트

## 7-1. 필수 로그 항목

아래 항목은 반드시 로그로 남긴다.

* 유저 ID
* 재화 종류
* 증감량
* 변경 전 수량
* 변경 후 수량
* 획득/소모 원인
* 관련 콘텐츠 ID
* 발생 시각

---

## 7-2. 운영 모니터링 포인트

### 1) 재화 인플레이션 체크

* 특정 기간 골드 총 생산량 / 총 소모량
* 다이아 무료 공급량 / 유료 공급량
* 소환권 배포량 추이

### 2) 재화 부족 구간 분석

* 어떤 재화에서 부족 팝업이 많이 뜨는지
* 어떤 성장 구간에서 막히는지
* 던전 티켓 부족 빈도

### 3) 획득처 이용률 분석

* 부족 팝업에서 어떤 바로가기 버튼이 많이 눌리는지
* 유저가 실제로 부족 재화 획득 루트를 따라가는지

### 4) 비정상 탐지

* 짧은 시간 내 비정상 대량 획득
* 음수/롤백 이력
* 동일 요청 반복 전송
* 충전형 재화 비정상 회복

---

## 7-3. 운영 액션 예시

* 골드 부족률이 지나치게 높으면 지역 보상량 상향
* 던전 티켓 부족 빈도가 높으면 자연 회복 속도 조정
* 특정 재화의 상세 팝업 클릭률이 낮으면 UI 가시성 개선
* 소환권 부족 이탈이 높으면 이벤트/퀘스트 공급선 추가

# 요약 기획 의도

재화 확인 시스템은 단순히 숫자를 보여주는 기능이 아니라,
**유저가 “지금 무엇이 부족한지”, “그걸 어디서 얻는지”, “현재 성장 흐름이 왜 막혔는지”를 즉시 이해하게 만드는 안내 시스템**이다.

이 시스템이 잘 설계되면:

* 유저 혼란이 줄고
* 이탈 구간을 완화할 수 있으며
* 각 콘텐츠로의 자연스러운 이동을 유도할 수 있다.
