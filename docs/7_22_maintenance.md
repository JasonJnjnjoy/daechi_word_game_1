# 점검 시스템 기획서

본 문서는 방치형 게임의 **점검 시스템**을 정의한다.
점검 시스템은 서버 안정화, 콘텐츠 반영, 밸런스 수정, 장애 대응을 위해 서비스 이용을 제한하거나 특정 기능을 일시 중단하는 운영 시스템이다.

이 프로젝트는 구조를 단순하게 유지하기 위해 **글로벌 단일 서버 점검**을 기본으로 하며, 채널별/서버군별 점검은 지원하지 않는다.

---

## 1. 시스템 개요

점검 시스템의 목적은 다음과 같다.

* 신규 콘텐츠, 데이터, 밸런스, 버그 수정 반영
* 장애 발생 시 빠른 서비스 차단 및 복구
* 유저에게 명확한 점검 일정과 상태 전달
* 점검 중 데이터 불일치 및 보상 누락 방지
* 점검 종료 후 안정적으로 서비스 재개

### 1-1. 점검 유형

#### 정기 점검

* 사전에 공지되는 점검
* 시작 시각과 종료 예정 시각이 존재
* 점검 전 카운트다운 공지 가능

#### 긴급 점검

* 장애, 악용, 치명 버그 대응용 점검
* 짧은 공지 또는 즉시 진입 가능
* 종료 시각이 미정일 수 있음

#### 부분 점검

* 전체 접속 차단 없이 특정 기능만 중단
* 예: 상점만 차단, 길드만 차단, 랭킹 집계만 중단
* 단, 본 프로젝트에서는 **전체 점검을 기본**으로 하고, 부분 점검은 운영 필요 시에만 사용

---

### 1-2. 점검 상태

점검 상태는 아래와 같이 정의한다.

| 상태        | 설명          |
| --------- | ----------- |
| normal    | 정상 운영 상태    |
| notice    | 점검 예정 공지 상태 |
| active    | 점검 진행 중     |
| extended  | 점검 연장 상태    |
| finished  | 점검 종료       |
| cancelled | 점검 취소       |

---

### 1-3. 설계 원칙

1. 점검 여부는 항상 **서버 기준**으로 판정한다.
2. 클라이언트 로컬 시간은 신뢰하지 않는다.
3. 점검 시작 시점 이후에는 신규 플레이 요청을 차단한다.
4. 점검 중에는 유저가 상태를 오해하지 않도록 명확한 안내 문구를 노출한다.
5. 점검 보상은 필요 시 **우편 시스템**을 통해 지급한다.
6. 방치형 게임 특성상 점검 시간은 **오프라인 보상 시간에 포함**된다. 단, 최대 12시간 상한은 유지한다.
7. 점검 종료는 예정 시각 자동 해제가 아니라, **운영자 확인 후 수동 종료**를 기본으로 한다.

---

## 2. 규칙 설명

## 2-1. 점검 진입 규칙

### 정기 점검

* 운영자가 점검 일정을 등록하면 `notice` 상태로 진입한다.
* 점검 시작 전 로비 배너, 팝업, 카운트다운 UI를 통해 공지한다.
* 예정 시각 도달 시 서버는 `active` 상태로 변경한다.
* `active` 전환 후 신규 로그인 및 주요 게임 액션을 차단한다.

### 긴급 점검

* 운영자가 즉시 `active` 상태로 전환할 수 있다.
* 긴급 점검은 공지 없이 바로 시작될 수 있으나, 가능한 경우 최소 1회 안내 메시지를 노출한다.
* 종료 시각이 없는 경우 “종료 시간 미정”으로 표기한다.

---

## 2-2. 점검 중 이용 제한 규칙

### 전체 점검 시

아래 기능은 모두 차단된다.

* 로그인
* 지역 전투 진입 및 진행
* 던전 입장
* 소환
* 성장/강화
* 상점 구매
* 길드 기능
* 랭킹 갱신
* 쿠폰 입력
* 우편 수령

### 부분 점검 시

* 점검 대상 기능만 차단한다.
* 차단되지 않은 기능은 정상 이용 가능하다.
* 부분 점검 여부와 차단 대상은 서버 설정값으로 판정한다.

---

## 2-3. 점검 시작 시 세션 처리 규칙

점검 시작 시 온라인 유저는 아래 순서로 처리한다.

1. 서버가 신규 액션 요청 접수 중단
2. 현재 접속 중인 유저에게 점검 시작 알림 전송
3. 유저 상태를 마지막 서버 기준 시점으로 저장
4. 게임 플레이를 종료시키고 점검 안내 화면으로 이동 또는 접속 종료
5. 점검 이후 재접속 시 점검 상태를 다시 확인

### 지역 전투 처리

* 일반 지역 전투는 마지막 서버 저장 시점을 기준으로 종료 처리한다.
* 점검 시작 직전까지 확정된 진행분만 인정한다.
* 점검 중 추가 전투 진행은 발생하지 않는다.
* 강제 종료 시점부터는 오프라인 상태로 간주한다.

### 오프라인 보상 처리

* 점검 중 접속하지 못하는 시간은 오프라인 누적 시간에 포함된다.
* 단, 오프라인 보상 최대치는 12시간으로 제한한다.

### 던전 처리

* 던전 입장 후 결과가 서버에 확정되기 전 점검이 시작된 경우, 해당 플레이는 **무효 처리**한다.
* 이 경우 소비된 던전 티켓은 환불한다.
* 던전 클리어 결과까지 서버 확정이 끝난 경우에는 정상 반영한다.

### 소환/강화 처리

* 서버 확정 응답 전 점검이 시작되면 해당 요청은 실패 처리한다.
* 서버 확정이 완료된 경우에는 정상 반영한다.
* 클라이언트와 서버 상태가 어긋난 경우 서버 상태를 기준으로 재동기화한다.

### 상점/결제 처리

* 점검 직전 결제가 진행 중인 경우, 결제 승인 여부와 지급 여부를 기준으로 복구 로직을 수행한다.
* 결제는 성공했으나 재화/상품이 지급되지 않은 경우, 운영 복구 대상 로그로 남긴다.
* 점검 중에는 신규 결제를 막는다.

---

## 2-4. 점검 공지 규칙

정기 점검의 경우 아래 공지 단계를 권장한다.

* 점검 등록 즉시: 공지 배너 노출
* 점검 30분 전: 로비 공지
* 점검 10분 전: 팝업 공지
* 점검 5분 전: 재공지
* 점검 1분 전: 최종 공지

긴급 점검의 경우:

* 즉시 전체 팝업 1회
* 로비 배너 및 점검 화면 노출
* 필요 시 채팅/공지 영역 반복 안내

---

## 2-5. 점검 종료 규칙

점검 종료는 예정 시간이 지났다고 자동으로 열리지 않는다.

운영자가 아래 항목을 확인한 뒤 `finished` 처리한다.

* 로그인 서버 정상
* 게임 서버 정상
* DB 연결 정상
* 주요 API 응답 정상
* 점검 대상 패치 반영 확인
* 오류율 정상 범위 확인

점검 종료 후:

* 로그인 재개
* 차단 API 해제
* 랭킹/길드/상점 등 필요한 기능 재활성화
* 필요 시 점검 보상 우편 발송
* 점검 완료 공지 노출

---

## 2-6. 점검 보상 규칙

점검 보상은 모든 점검에 자동 지급되지 않는다.
운영 판단에 따라 지급 여부를 결정한다.

### 보상 지급 대상

기본 권장 조건:

* 점검 시작 시각 이전에 생성된 계정
* 제재 계정 제외
* 동일 점검 보상은 계정당 1회만 지급

### 보상 지급 방식

* 우편 시스템을 통해 지급
* 보상 종류는 현재 게임의 6종 재화만 허용

  * 골드
  * 다이아
  * 장비 소환권
  * 스킬 소환권
  * 던전 티켓
  * 길드 코인

### 보상 지급 시점

* 점검 종료 직후 일괄 발송
* 또는 운영자가 수동 발송

---

## 2-7. 클라이언트 업데이트 연계 규칙

점검 이후 강제 업데이트가 필요한 경우, 서버에서 `required_client_version`을 내려준다.

* 클라이언트 버전이 기준 미만이면 로그인 차단
* 점검 종료 화면 대신 업데이트 유도 화면 노출
* 업데이트 완료 후 재접속 가능

---

## 3. 데이터 구조

점검 시스템은 **점검 마스터**, **차단 범위**, **보상 데이터**, **운영 이력**, **보상 지급 이력**으로 구성한다.

## 3-1. 점검 마스터

### `maintenance_master`

| 필드명                     | 타입                   | 설명                                                         |
| ----------------------- | -------------------- | ---------------------------------------------------------- |
| maintenance_id          | bigint PK            | 점검 ID                                                      |
| title                   | varchar(100)         | 점검 제목                                                      |
| description             | text                 | 점검 설명                                                      |
| maintenance_type        | enum                 | scheduled / emergency                                      |
| scope_type              | enum                 | full / partial                                             |
| status                  | enum                 | normal / notice / active / extended / finished / cancelled |
| notice_start_at         | datetime nullable    | 사전 공지 시작 시각                                                |
| start_at                | datetime             | 점검 시작 예정 시각                                                |
| end_at                  | datetime nullable    | 점검 종료 예정 시각                                                |
| actual_start_at         | datetime nullable    | 실제 시작 시각                                                   |
| actual_end_at           | datetime nullable    | 실제 종료 시각                                                   |
| required_client_version | varchar(20) nullable | 점검 후 최소 요구 클라 버전                                           |
| allow_gm_login          | bool                 | 운영/GM 접속 허용 여부                                             |
| created_by              | bigint               | 생성 운영자 ID                                                  |
| updated_by              | bigint               | 수정 운영자 ID                                                  |
| created_at              | datetime             | 생성 시각                                                      |
| updated_at              | datetime             | 수정 시각                                                      |

---

## 3-2. 점검 차단 범위

부분 점검을 지원하기 위한 테이블이다.

### `maintenance_scope_detail`

| 필드명             | 타입                    | 설명                                                                        |
| --------------- | --------------------- | ------------------------------------------------------------------------- |
| scope_detail_id | bigint PK             | 상세 ID                                                                     |
| maintenance_id  | bigint FK             | 점검 ID                                                                     |
| feature_code    | varchar(32)           | login / stage / dungeon / summon / shop / guild / ranking / mail / coupon |
| block_mode      | enum                  | block / readonly / hide                                                   |
| message         | varchar(255) nullable | 해당 기능 차단 안내 문구                                                            |

전체 점검 시에는 별도 상세 없이 `scope_type = full`만으로 처리할 수 있다.

| maintenance_id        | bigint FK   | 점검 ID                                                                      |
| reward_type           | varchar(32) | gold / diamond / equip_ticket / skill_ticket / dungeon_ticket / guild_coin |

## 4. UI 구성

## 4-1. 사전 공지 UI

### 로비 배너

* 점검 예정 시간 표시
* 점검 제목 및 간단한 사유 표시
* 클릭 시 상세 팝업 진입

### 점검 상세 팝업

표시 항목:

* 점검 제목
* 점검 시작 시각
* 점검 종료 예정 시각
* 점검 사유
* 영향 범위
* 보상 지급 여부 안내(optional)

---

## 4-2. 임박 안내 UI

점검 10분 전부터는 팝업 노출을 권장한다.

표시 내용:

* “잠시 후 점검이 시작됩니다.”
* 남은 시간 카운트다운
* 진행 중 콘텐츠가 종료될 수 있다는 안내

점검 1분 전에는 강한 경고 형태로 1회 더 노출한다.

---

## 4-3. 점검 시작 시 UI

### 강제 종료 팝업

* “점검이 시작되어 게임 이용이 종료됩니다.”
* 확인 버튼
* 확인 후 타이틀 또는 점검 전용 화면 이동

### 점검 전용 화면

표시 항목:

* 점검 중 문구
* 점검 사유
* 종료 예정 시간 또는 “종료 시간 미정”
* 점검 완료 후 다시 접속해 달라는 안내
* 필요 시 업데이트 버튼 노출

---

## 4-4. 부분 점검 UI

부분 점검 시에는 전체 화면 차단 대신 대상 기능에만 아래 형태 적용

* 버튼 비활성화
* “현재 점검 중인 기능입니다” 툴팁
* 해당 시스템 진입 차단 팝업

예시:

* 상점만 점검 중이면 상점 버튼 비활성화
* 랭킹만 점검 중이면 랭킹 화면에서 “집계 중” 문구 표시

---

## 4-5. 점검 종료 후 UI

### 접속 재개 시

* “점검이 완료되었습니다.” 팝업 1회 노출
* 점검 보상이 있는 경우 “우편함을 확인해 주세요” 안내
* 강제 업데이트가 필요한 경우 업데이트 화면 우선 노출

## 5. 예외 처리

| 상황                           | 처리 방식                            |
| ---------------------------- | -------------------------------- |
| 점검 예정 시간이 지났지만 실제 오픈 준비가 안 됨 | `extended` 상태로 전환, 새 종료 예정 시각 공지 |
| 점검 중 종료 시각 미정                | 점검 화면에 “종료 시간 미정” 표시             |
| 점검 도중 운영자가 취소                | `cancelled` 처리 후 즉시 서비스 복구       |
| 유저가 던전 플레이 중 점검 시작           | 결과 미확정이면 무효 처리 + 티켓 환불           |
| 소환/강화 요청 도중 점검 시작            | 서버 확정 여부 기준으로 성공/실패 결정           |
| 결제 승인 후 지급 누락                | 결제 복구 로그에 적재, 운영 복구 대상 처리        |
| 점검 중 반복 재접속 시도               | 점검 화면만 반환, 로그인 토큰 갱신 차단 가능       |
| 종료 예정 시간과 실제 종료 시간이 다름       | 실제 종료 시각을 별도 기록                  |
| 점검 종료 후 클라이언트 구버전 접속         | 강제 업데이트 화면 노출                    |
| 부분 점검 중 차단 기능 접근             | 기능별 차단 팝업 노출                     |
| 보상 우편 중복 발송                  | 지급 이력 기준 계정당 1회만 발송              |
| 장시간 점검으로 오프라인 보상 상한 초과       | 최대 12시간까지만 인정                    |

### 추가 원칙

* 점검 중에는 서버가 유저 요청을 “성공처럼 보이게” 처리하면 안 된다.
* 점검으로 인해 유저 자산 손실이 발생하면 안 된다.
* 보상, 결제, 티켓, 소환 결과는 반드시 서버 확정 여부를 기준으로 처리한다.

## 7-3. 운영 포인트

### 운영 체크리스트

점검 시작 전:

* 점검 공지 등록
* 점검 범위 및 보상 여부 확정
* 강제 업데이트 여부 확정
* DB 백업 및 배포 파일 확인
* 점검 후 검증 항목 준비

점검 중:

* 로그인/API 차단 상태 확인
* 배포 및 데이터 반영
* 오류 로그 모니터링
* 필요 시 종료 시간 연장 공지

점검 종료 전:

* 로그인 테스트
* 전투/던전/소환/상점/길드 핵심 흐름 테스트
* 재화 지급/저장/우편 테스트
* 오류율 정상 확인
* 보상 발송 준비

점검 종료 후:

* 완료 공지 노출
* 보상 발송
* 초기 30분 집중 모니터링

## 7-5. 간단 정책 요약

* 점검 판정은 서버 기준
* 전체 점검이 기본, 부분 점검은 선택
* 점검 시간은 오프라인 보상 시간에 포함
* 던전 미확정 결과는 무효 + 티켓 환불
* 점검 종료는 자동이 아닌 운영 확인 후 수동
* 점검 보상은 우편으로 계정당 1회 지급

다음 시스템 기획서를 이어서 작성할 때는 **공지**, **우편**, **이벤트**, **오프라인 보상**과 특히 연결이 잘 됩니다.
