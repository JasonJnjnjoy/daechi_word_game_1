# 쿠폰 시스템 기획서

본 문서는 방치형 게임의 **쿠폰 시스템**을 정의한다.
쿠폰 시스템은 운영 보상, 사전예약 보상, 이벤트 보상, 점검 보상, 제휴 보상 등을 유저에게 빠르게 지급하기 위한 기능이다.
본 프로젝트에서는 구조를 단순화하기 위해 **쿠폰 보상은 현재 재화 6종만 지급 가능**하도록 설계한다.

---

## 1. 시스템 개요

쿠폰 시스템은 유저가 특정 코드를 입력하면 서버 검증 후 보상을 즉시 수령하는 시스템이다.
운영 목적은 다음과 같다.

* 이벤트/프로모션 보상 지급
* 점검 보상/장애 보상 지급
* 사전예약/공식 채널 보상 지급
* 특정 유저군 대상 보상 지급

### 1-1. 설계 목표

* 입력과 수령 절차를 단순하게 유지
* 보상 중복 수령 방지
* 운영툴에서 생성/비활성화/연장 가능
* 서버 기준으로 유효성 판정
* 악용 가능성 최소화

### 1-2. 지원 범위

현재 쿠폰 시스템은 다음 보상만 지원한다.

* 골드
* 다이아
* 장비 소환권
* 스킬 소환권
* 던전 티켓
* 길드 코인

장비, 스킬, 캐릭터, 버프, 우편 전용 아이템 등은 현재 버전에서 지원하지 않는다.

### 1-3. 쿠폰 유형

운영 편의를 위해 쿠폰은 아래 3종만 지원한다.

1. **공용 쿠폰**

   * 모든 유저가 같은 코드 입력
   * 계정당 1회 사용 가능

2. **조건부 공용 쿠폰**

   * 같은 코드 사용
   * 특정 조건 충족 유저만 사용 가능
   * 예: 계정 생성 7일 이상, 특정 지역 진입 이상, 길드 가입 상태

3. **개별 쿠폰**

   * 유저마다 다른 고유 코드
   * 특정 계정에만 사용 가능

## 2. 규칙 설명

## 2-1. 기본 규칙

1. 쿠폰은 **계정 단위**로 사용 이력을 관리한다.
2. 같은 쿠폰은 동일 계정에서 **1회만 사용 가능**하다.
3. 쿠폰 유효성은 **서버 시간 기준**으로 판정한다.
4. 보상은 검증 성공 즉시 지급한다.
5. 쿠폰 입력 성공 시 보상은 즉시 wallet에 반영한다.
6. 쿠폰 보상이 재화 상한을 초과하는 경우 정책에 따라 지급/우편 전환/실패 처리한다.
7. 게스트 계정은 쿠폰 사용 불가로 운영하는 것을 기본 정책으로 한다.
8. 영문/숫자 조합의 코드만 허용하며, 입력 시 대소문자는 구분하지 않는다.

---

## 2-2. 쿠폰 사용 흐름

1. 유저가 쿠폰 입력 화면에서 코드 입력
2. 클라이언트가 서버에 쿠폰 사용 요청 전송
3. 서버가 아래 항목 검증

   * 코드 존재 여부
   * 활성화 여부
   * 시작일/종료일
   * 전체 사용 가능 횟수
   * 계정별 사용 여부
   * 대상 계정 조건 충족 여부
4. 검증 성공 시 보상 지급
5. 사용 이력 저장
6. 결과 반환
7. 클라이언트가 보상 팝업 노출

---

## 2-3. 쿠폰 검증 규칙

### 입력 규칙

* 허용 길이: 6~20자
* 허용 문자: A-Z, 0-9
* 공백 입력 시 trim 처리
* 특수문자는 허용하지 않음

### 상태 규칙

쿠폰은 아래 상태 중 하나를 가진다.

* `scheduled` : 시작 전
* `active` : 사용 가능
* `expired` : 만료
* `disabled` : 운영 중지
* `exhausted` : 전체 사용 횟수 소진

### 수령 규칙

* 1계정 1회 수령
* 중복 요청은 `request_id` 기준으로 1회만 인정
* 지급과 사용 이력 저장은 원자적으로 처리
* 부분 지급은 허용하지 않음

---

## 2-4. 지급 보상 규칙

쿠폰 1개는 복수 재화를 동시에 지급할 수 있다.

예시:

* 골드 10000 + 다이아 300
* 장비 소환권 10 + 스킬 소환권 10
* 던전 티켓 5 + 길드 코인 300

보상 지급 원칙:

* 재화 단위는 모두 정수
* 음수 지급 불가
* 0 이하 값 불가
* 미정의 재화 코드 불가

---

## 2-5. 조건부 쿠폰 규칙

조건부 쿠폰은 아래 조건 중 필요한 항목만 선택해서 설정할 수 있다.

* 최소 계정 생성일 경과
* 최소 지역 진입 단계
* 최소 레벨
* 길드 가입 여부
* 특정 이벤트 참여 여부
* 특정 플랫폼 전용 여부 (AOS / iOS)

현재 버전에서는 지나친 복잡도를 피하기 위해 조건식을 자유 입력하지 않고, **미리 정한 조건 타입만 선택** 가능하게 한다.

---

## 2-6. 운영 정책 권장안

### 추천 정책

* 보상 쿠폰은 대부분 **7일~30일 유효기간**
* 사전예약/오픈 기념 쿠폰은 **30일 이상**
* 점검 보상 쿠폰은 **3일~7일**
* 전체 사용 횟수 제한은 특별한 이유가 없으면 두지 않음
* 공식 채널 공개 쿠폰은 계정당 1회만 허용

## 3. 데이터 구조

쿠폰 시스템은 **쿠폰 마스터**, **보상 데이터**, **사용 이력**, **요청 중복 방지 로그**로 구성한다.

## 3-1. 쿠폰 마스터

### `coupon_master`

| 필드명                  | 타입                 | 설명                                                  |
| -------------------- | ------------------ | --------------------------------------------------- |
| coupon_id            | bigint PK          | 쿠폰 ID                                               |
| coupon_code          | varchar(32) UNIQUE | 실제 입력 코드                                            |
| coupon_name          | varchar(100)       | 운영용 이름                                              |
| coupon_type          | enum               | common / conditional / unique                       |
| status               | enum               | scheduled / active / expired / disabled / exhausted |
| start_at             | datetime           | 사용 시작 시각                                            |
| end_at               | datetime           | 사용 종료 시각                                            |
| max_total_use_count  | int nullable       | 전체 사용 가능 횟수                                         |
| current_use_count    | int                | 현재 사용 횟수                                            |
| target_user_id       | bigint nullable    | 개별 쿠폰 대상 계정                                         |
| min_account_age_day  | int nullable       | 최소 계정 생성일 조건                                        |
| min_stage_id         | int nullable       | 최소 지역/스테이지 조건                                       |
| min_user_level       | int nullable       | 최소 레벨 조건                                            |
| require_guild_joined | bool               | 길드 가입 필요 여부                                         |
| platform_limit       | enum nullable      | all / aos / ios                                     |
| created_by           | bigint             | 생성 운영자                                              |
| created_at           | datetime           | 생성 시각                                               |
| updated_at           | datetime           | 수정 시각                                               |

---

## 3-2. 쿠폰 보상 데이터

### `coupon_reward`

| 필드명              | 타입          | 설명                                                                         |
| ---------------- | ----------- | -------------------------------------------------------------------------- |
| coupon_reward_id | bigint PK   | 보상 ID                                                                      |
| coupon_id        | bigint FK   | 쿠폰 ID                                                                      |
| reward_type      | varchar(32) | gold / diamond / equip_ticket / skill_ticket / dungeon_ticket / guild_coin |
| reward_amount    | bigint      | 지급 수량                                                                      |

쿠폰 1개가 여러 보상을 가질 수 있으므로 1:N 구조로 둔다.

---

## 3-3. 쿠폰 사용 이력

### `coupon_redeem_log`

| 필드명                  | 타입                    | 설명             |
| -------------------- | --------------------- | -------------- |
| redeem_id            | bigint PK             | 사용 이력 ID       |
| coupon_id            | bigint                | 쿠폰 ID          |
| user_id              | bigint                | 사용 유저 ID       |
| coupon_code_snapshot | varchar(32)           | 사용 당시 코드       |
| result               | enum                  | success / fail |
| fail_reason          | varchar(32) nullable  | 실패 사유          |
| request_id           | varchar(64)           | 중복 요청 방지 키     |
| redeemed_at          | datetime              | 사용 시각          |
| client_platform      | enum                  | aos / ios / pc |
| client_version       | varchar(20)           | 클라 버전          |
| ip_hash              | varchar(128) nullable | 보안/운영 추적용      |
| device_hash          | varchar(128) nullable | 보안/운영 추적용      |

성공/실패 모두 남기는 것을 권장한다.

---

## 3-4. 재화 지급 로그 연동

쿠폰 사용 성공 시 기존 재화 로그에도 기록한다.

### `currency_tx_log`

`reason_code = COUPON_REWARD`

예시:

* user_id: 101
* currency_code: diamond
* change_amount: +300
* reason_code: COUPON_REWARD
* reason_ref_id: coupon_id

## 4. UI 구성

## 4-1. 진입 위치

쿠폰 UI는 설정 화면 또는 이벤트 메뉴 내 서브 버튼으로 진입한다.

권장 위치:

* 설정 > 쿠폰 등록
  또는
* 이벤트 > 쿠폰 입력

복잡도를 줄이기 위해 메인 화면 고정 버튼은 두지 않는다.

---

## 4-2. 쿠폰 입력 화면 구성

구성 요소:

* 화면 제목: 쿠폰 등록
* 1줄 입력창
* 등록 버튼
* 사용 안내 문구
* 최근 등록 성공 보상 미리보기 영역(optional)

### 입력창 규칙

* 최대 20자
* 영어 대문자/숫자 자동 정리
* 붙여넣기 지원
* 입력 중 공백 자동 제거

### 안내 문구 예시

* 쿠폰은 계정당 1회만 사용할 수 있습니다.
* 보상은 즉시 지급됩니다.
* 유효기간이 지난 쿠폰은 사용할 수 없습니다.

---

## 4-3. 성공 결과 화면

등록 성공 시 팝업 표시:

* “쿠폰이 등록되었습니다.”
* 지급된 재화 리스트
* 확인 버튼

예시:

* 골드 +10,000
* 다이아 +300
* 장비 소환권 +5

---

## 4-4. 실패 메시지 구성

실패 사유는 모호하지 않게 분리한다.

| 실패 상황      | 메시지 예시                  |
| ---------- | ----------------------- |
| 존재하지 않는 쿠폰 | 유효하지 않은 쿠폰입니다.          |
| 이미 사용한 쿠폰  | 이미 사용한 쿠폰입니다.           |
| 기간 만료      | 사용 기간이 종료된 쿠폰입니다.       |
| 시작 전       | 아직 사용할 수 없는 쿠폰입니다.      |
| 조건 미충족     | 사용 조건을 충족하지 않은 쿠폰입니다.   |
| 대상 계정 아님   | 이 계정에서는 사용할 수 없는 쿠폰입니다. |
| 사용 횟수 소진   | 사용 가능 횟수가 모두 소진된 쿠폰입니다. |
| 서버 오류      | 잠시 후 다시 시도해 주세요.        |

---

## 4-5. UX 보조 요소

* 쿠폰 성공 시 즉시 현재 재화 수량 갱신
* 네트워크 중 중복 입력 방지를 위해 버튼 로딩 상태 표시
* 붙여넣기 사용이 많으므로 모바일 키보드 최적화
* 잘못된 문자는 입력 단계에서 차단

## 5. 예외 처리

| 상황                 | 처리 방식                    |
| ------------------ | ------------------------ |
| 동일 요청 연속 전송        | `request_id` 기준 1회만 처리   |
| 쿠폰 입력 후 네트워크 끊김    | 서버 처리 결과 기준으로 재조회        |
| 종료 시각 직전 요청        | 서버 수신 시각 기준 판정           |
| 지급 중 서버 오류         | 전체 롤백, 성공 이력 저장 금지       |
| 보상 지급 후 이력 저장 실패   | 전체 트랜잭션 롤백               |
| 코드 앞뒤 공백 포함        | trim 후 검증                |
| 소문자 입력             | 대문자로 정규화 후 검증            |
| 존재하지만 비활성 상태       | 상태별 실패 메시지 반환            |
| 개별 쿠폰을 타 계정이 사용 시도 | 실패 처리 및 보안 로그 기록         |
| 보상 상한 초과           | 정책에 따라 지급 제한/우편 전환/실패 처리 |

### 추가 원칙

* 쿠폰 사용은 반드시 **트랜잭션 단위**로 처리한다.
* 지급 성공 후 사용 이력 미기록 상태는 절대 허용하지 않는다.
* 운영자가 쿠폰을 비활성화한 이후에는 즉시 사용 불가 상태가 되어야 한다.

## 7. 로그 및 운영 포인트

## 7-1. 필수 로그

### 성공 로그

* 어떤 쿠폰이 사용되었는지
* 누가 사용했는지
* 언제 사용했는지
* 어떤 보상이 지급되었는지
* 어떤 기기/플랫폼에서 사용했는지

### 실패 로그

* 어떤 코드가 입력되었는지
* 실패 사유가 무엇인지
* 동일 계정/디바이스/IP에서 반복 시도되는지

---

## 7-2. 핵심 운영 지표

1. 쿠폰별 총 사용 수
2. 쿠폰별 성공률 / 실패율
3. 실패 사유 분포
4. 채널별 회수율

   * 공지
   * 디스코드
   * 카페
   * 인게임 배너
5. 쿠폰 등록 후 D1/D3/D7 재접속률
6. 쿠폰 사용 유저의 평균 성장 구간
7. 보상 지급 후 재화 인플레이션 영향

## 7-3. 운영 포인트

### 운영자가 자주 조정하는 항목

* 시작/종료 일시
* 보상 수량
* 사용 가능 대상
* 최대 사용 가능 횟수
* 상태 변경(active/disabled)

### 추천 운영 활용 방식

* 점검 보상: 짧은 유효기간 + 전 유저 공용
* 채널 팔로우 유도: 공식 SNS 공개 쿠폰
* 복귀 유도: 특정 기간 미접속 유저 전용 개별 쿠폰
* CS 보상: 개별 쿠폰 발급
* 제휴 프로모션: 조건부 공용 쿠폰

## 7-4. 악용 대응 포인트

* 동일 IP/디바이스에서 다계정 반복 사용 패턴 탐지
* 짧은 시간 내 실패 입력 과다 발생 시 rate limit
* 관리자 툴에서 쿠폰 즉시 비활성화 기능 제공
* 개별 쿠폰 유출 여부 추적 가능하도록 발급 배치 정보 저장 권장
