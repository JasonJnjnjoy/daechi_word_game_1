좋아. 아래는 네가 제시한 제약 조건을 기준으로 정리한 **전투 시스템 기획서**야.
기본 전제는 **완전 자동 전투**, **단일 PvE 중심**, **직접 조작 없는 방치형 전투**다.

# 전투 시스템 기획서

## 1. 시스템 개요

### 1-1. 목적

전투 시스템은 플레이어가 현재 선택한 지역에서 자동으로 몬스터를 처치하고, 그 결과로 **골드 획득**, **지역 진행**, **오프라인 보상 기준 갱신**을 수행하는 핵심 시스템이다.
전투 자체는 조작의 재미보다 **성장 체감**과 **반복 플레이의 효율감**을 제공하는 데 목적이 있다.

### 1-2. 전투의 역할

전투는 전체 코어 루프에서 아래 역할을 담당한다.

* 현재 지역의 몬스터 자동 처치
* 골드 수급의 기본 축
* 지역/스테이지 돌파
* 보스 클리어를 통한 다음 구간 해금
* 오프라인 보상 계산 기준 제공
* 장비/스킬/성장 강화의 효과 체감 제공

### 1-3. 설계 방향

이 게임의 전투는 아래 방향으로 설계한다.

* **완전 자동 전투**
* **단순한 수치 비교 중심**
* **직관적인 진행 구조**
* **낮은 서버 비용**
* **치트 방지를 위한 서버 권한 유지**
* **성장 시스템과 강하게 연결**

### 1-4. 전투 기본 구조

전투는 다음 흐름으로 진행된다.

1. 플레이어가 지역/스테이지에 진입
2. 일반 몬스터가 순차적으로 등장
3. 플레이어가 기본 공격 + 장착 스킬 자동 사용
4. 일반 몬스터를 일정 수 처치하면 보스 등장
5. 보스 처치 시 다음 스테이지 진입
6. 실패 시 마지막 클리어 지점 기준으로 재도전 또는 자동 파밍 유지

### 1-5. 기본 전제

본 기획에서는 구현 단순화를 위해 다음을 기본값으로 둔다.

* 전투는 **1인 PvE**
* 적은 **동시에 1마리만 전투**
* 플레이어 조작은 없음
* 전투 스킬은 **프리셋에 등록된 스킬만 자동 사용**
* 일반 필드 보상은 **골드 중심**
* 최초 보스 클리어 시에만 추가 보상 가능
* 오프라인 보상 최대 누적 시간은 **12시간**

## 2. 규칙 설명

## 2-1. 진입 규칙

* 플레이어는 해금된 지역/스테이지만 진입 가능하다.
* 현재 선택한 지역 내에서 전투가 자동 시작된다.
* 이전 지역으로 돌아가 파밍할 수 있다.
* 가장 최근에 선택한 지역/스테이지를 다음 접속 시 기본 전투 위치로 사용한다.

## 2-2. 전투 단위

전투는 **스테이지 단위**로 관리한다.

예시 기본값:

* 일반 몬스터 10마리 처치
* 이후 보스 1마리 등장
* 보스 클리어 시 다음 스테이지 이동

지역은 여러 개의 스테이지로 구성된다.
예: 지역 1 = 스테이지 1~10, 지역 2 = 스테이지 11~20

## 2-3. 전투 방식

* 플레이어는 자동으로 기본 공격을 수행한다.
* 장착한 스킬은 자동 발동된다.
* 기본 공격과 스킬은 모두 쿨타임 또는 공격속도 규칙을 따른다.
* 적 처치 시 즉시 다음 적이 등장한다.
* 별도 이동 조작은 없다.

## 2-4. 스킬 사용 규칙

전투 시스템은 #21 스킬 프리셋에 저장된 장착 스킬 정보를 참조한다.

* 전투 중 사용 가능한 스킬은 현재 프리셋 기준
* 스킬은 조건 충족 시 자동 발동
* 동시에 여러 스킬이 발동 가능 상태일 경우 **우선순위 규칙** 적용
* 우선순위는 기본적으로 **프리셋 슬롯 순서 기준**
* 전투 중 프리셋 변경 시 즉시 반영하지 않고 **다음 전투 시작 시점** 또는 **현재 적 처치 후 반영**

권장 단순 규칙:

* 스킬 타입은 우선 3종만 지원

  * 공격형
  * 버프형
  * 회복/보호형

## 2-5. 데미지 규칙

전투 수식은 단순해야 한다.

권장 기본식:

* 기본 공격 데미지
  `최종 데미지 = max(1, 공격력 - 대상 방어력)`
* 스킬 데미지
  `최종 데미지 = max(1, (공격력 × 스킬계수) - 대상 방어력)`
* 치명타 발생 시
  `최종 데미지 = 최종 데미지 × 치명타 배율`

추가 고려 가능 항목:

* 공격속도
* 치명타 확률
* 치명타 배율
* 보스 추가 피해
* 스킬 피해 증가

단, 초기 버전은 복잡도를 줄이기 위해 상태이상은 최소화하는 것을 권장한다.

## 2-6. 승패 규칙

### 일반 몬스터 전투

* 몬스터 HP가 0이 되면 플레이어 승리
* 플레이어 HP가 0이 되면 패배
* 패배 시 일정 시간 후 자동 부활 후 재전투

### 보스 전투

* 제한 시간 내 보스 처치 시 클리어
* 플레이어 사망 또는 시간 초과 시 실패
* 실패 시 해당 보스 스테이지 클리어는 인정되지 않음
* 실패 후에는 **마지막으로 클리어한 스테이지 기준**으로 자동 전투 복귀

권장 기본값:

* 보스 제한 시간: 30초
* 사망 후 부활 대기: 3초

## 2-7. 지역 진행 규칙

* 스테이지 보스를 클리어하면 다음 스테이지 해금
* 특정 스테이지 클리어 시 다음 지역 해금
* 자동 진행 ON 상태에서는 클리어 즉시 다음 스테이지로 이동
* 자동 진행 OFF 상태에서는 현재 스테이지 반복 파밍

## 2-8. 보상 규칙

전투 보상은 최대한 단순하게 유지한다.

### 일반 몬스터 보상

* 골드 지급

### 보스 보상

* 골드 대량 지급
* 최초 클리어 시 다이아 또는 1회성 보상 지급 가능
* 최고 클리어 기록 갱신

### 오프라인 보상 기준

* 오프라인 보상은 **마지막으로 안정적으로 파밍 가능한 스테이지**를 기준으로 계산
* 보스 미클리어 진행 중 종료한 경우, 마지막 클리어 스테이지 기준 정산

## 2-9. 반복 파밍 규칙

전투가 불가능할 정도로 스테이지 난도가 높다면, 플레이어는 해당 구간에서 정체된다.
이 경우 시스템은 강제로 진행시키지 않고, 마지막 클리어 지점에서 계속 골드를 수급하게 하여 성장 후 재도전하게 만든다.

## 4. UI 구성

## 4-1. 메인 전투 화면

전투 화면은 가능한 한 단순하게 구성한다.

### 상단 영역

* 현재 지역명
* 현재 스테이지
* 진행도 표시
  예: `일반 7/10` 또는 `BOSS`
* 자동 진행 ON/OFF 버튼
* 현재 획득 골드 표시(간략)

### 중앙 영역

* 플레이어 캐릭터
* 현재 적 몬스터
* 플레이어 HP 바
* 몬스터 HP 바
* 보스 등장 시 보스 전용 UI
* 보스 제한 시간 바

### 하단 영역

* 장착 스킬 아이콘
* 각 스킬 쿨타임 오버레이
* 현재 적용 프리셋 표시
* 전투 연출 이펙트 영역

### 보조 영역

* 스테이지 클리어 토스트
* 보스 등장 경고
* 패배/부활 표시
* 획득 골드 플로팅 텍스트

* 현재 지역/스테이지 정보 표시
* 다음 지역 진입 가능 여부 표시
* 막힌 구간일 경우 “성장 필요” 안내 문구 출력

## 4-3. 전투 상태별 UI

### 일반 전투

* 몬스터명
* HP 바
* 진행도

### 보스 전투

* 보스 이름
* 대형 HP 바
* 남은 시간
* 경고 연출

* 부활 카운트
* 자동 재도전 안내

## 4-4. 사용자 조작 요소

직접 전투 조작은 없고, 아래 정도만 둔다.

* 자동 진행 ON/OFF
* 지역 이동
* 프리셋 선택
* 스킬 정보 확인
* 전투 속도 옵션은 초기 버전에서는 제외 가능

## 5. 예외 처리

## 5-1. 접속 종료/앱 백그라운드

* 전투 중 앱 종료 시 서버는 종료 시각을 기록
* 이후 접속 시 오프라인 보상으로 정산
* 오프라인 보상은 최대 12시간까지만 인정

## 5-2. 보스 진행 중 종료

* 보스 전투 중 접속 종료하더라도 보스 클리어는 보장하지 않음
* 마지막 안정 클리어 스테이지 기준으로 오프라인 보상 정산

## 5-3. 전투 도중 프리셋 변경

* 실시간 전투 도중 프리셋 변경 시 현재 타겟 전투 종료 후 반영
* 보스전 중에는 즉시 반영하지 않는 것을 권장
* 악용 방지를 위해 전투 스냅샷 기준을 유지

## 5-4. 스탯 비정상 값

* 공격력, 체력, 방어력 등 주요 스탯이 0 이하일 경우 최소값 보정
* 비정상 데이터 감지 시 해당 전투 정산 실패 처리 및 재동기화

## 5-5. 중복 정산 요청

* 동일 세션에 대해 여러 번 정산 요청이 들어오면 서버는 최초 1회만 유효 처리
* snapshot_id 또는 settlement_id 기반 idempotency 처리 필요

## 5-6. 통신 끊김

* 클라이언트 연출이 끊겨도 서버 기준 전투 상태는 유지
* 재접속 시 서버 스냅샷 기준으로 전투 상태 복원
* 클라이언트 연출 결과와 서버 결과가 다르면 서버 결과 우선

## 5-7. 다중 접속

* 동일 계정 다중 접속 시 마지막 접속 세션만 유효
* 이전 세션은 강제 종료 또는 관전 불가 처리

## 5-8. 진행 불가 상태

* 현재 선택한 지역이 아직 해금되지 않았거나 데이터 꼬임이 발생한 경우
* 서버는 자동으로 마지막 클리어 지역/스테이지로 롤백

---

## 6. 서버/클라이언트 처리 구분

이 게임은 방치형이므로 **실시간 프레임 단위 서버 전투**보다는 **시간 구간 정산 방식**이 적합하다.

## 6-1. 서버 처리

서버가 권한을 가져야 하는 항목:

* 플레이어 최종 능력치 계산
* 현재 스테이지 정보 저장
* 최고 클리어 기록 저장
* 전투 결과 정산
* 골드 지급
* 보스 클리어 판정
* 오프라인 보상 계산
* 중복 정산 방지
* 비정상 패턴 탐지

즉, **전투 결과와 보상은 반드시 서버 확정**이어야 한다.

## 6-2. 클라이언트 처리

클라이언트가 담당하는 항목:

* 전투 연출
* 캐릭터/몬스터 애니메이션
* 데미지 숫자 표시
* 스킬 이펙트 표시
* HP 바 연출
* 보스 등장 연출
* UI 버튼 처리
* 서버 정산 결과의 시각화

즉, **보여주는 전투는 클라이언트**, **실제 결과는 서버** 구조다.

## 6-3. 권장 처리 방식

### 추천 구조: 구간 정산형 전투

1. 전투 시작 시 서버가 전투 스냅샷 생성
2. 클라이언트는 이를 바탕으로 전투를 연출
3. 일정 시간마다 또는 특정 이벤트마다 서버에 정산 요청
4. 서버는 지난 시간 동안의 처치 수, 보스 결과, 골드 획득량 계산
5. 결과를 반환하고 클라이언트는 화면만 갱신

권장 정산 시점:

* 5~10초 주기
* 스테이지 클리어 시
* 보스 전투 종료 시
* 앱 종료/백그라운드 전환 시
* 재접속 시

이 구조의 장점:

* 서버 부하 절감
* 치트 대응 가능
* 방치형 특성에 적합
* 오프라인 보상과 같은 계산 로직 재사용 가능

## 6-4. 서버 계산 기준

서버는 아래 요소를 기준으로 정산한다.

* 전투 시작 시점 능력치 스냅샷
* 장착 스킬 프리셋
* 스테이지 난이도
* 지난 정산 이후 경과 시간
* 보스 도달 여부 및 클리어 가능 여부

## 6-5. 클라이언트 예측 허용 범위

클라이언트는 자연스러운 화면을 위해 예측 연출을 할 수 있다.
하지만 아래는 클라이언트 확정 불가다.

* 골드 획득 확정
* 보스 클리어 확정
* 다음 스테이지 진입 확정
* 최고 기록 갱신
* 오프라인 보상 확정

## 7. 로그 및 운영 포인트

## 7-1. 필수 로그

운영 및 밸런스 확인을 위해 최소 아래 로그가 필요하다.

* battle_start
* battle_settlement
* monster_kill
* boss_spawn
* boss_clear
* boss_fail
* stage_enter
* stage_clear
* player_death
* skill_preset_change
* offline_settlement
* reconnect_recovery

## 7-2. 주요 로그 항목

각 로그에는 아래 정보가 포함되어야 한다.

* user_id
* area_id
* stage_id
* snapshot_id
* selected_skill_preset_id
* player_power
* elapsed_time
* earned_gold
* death_count
* boss_result
* highest_clear_stage_before/after
* client_version

## 7-3. 운영 지표

운영에서는 아래 지표를 특히 봐야 한다.

### 진행 관련

* 스테이지별 평균 체류 시간
* 보스 클리어율
* 특정 구간 이탈률
* 지역별 도달 유저 비율

### 경제 관련

* 스테이지별 분당 골드 생산량
* 오프라인 보상 평균 수령량
* 유저 구간별 재화 축적 속도

### 전투 관련

* 평균 사망 횟수
* 스킬 프리셋 사용률
* 자동 진행 ON 비율
* 보스 실패 후 재도전 비율

## 7-4. 밸런싱 운영 포인트

운영자가 자주 조정하게 될 가능성이 높은 값은 다음과 같다.

* 일반 몬스터 HP 증가율
* 보스 HP/공격력 증가율
* 스테이지 골드 계수
* 보스 제한 시간
* 최초 클리어 보상량
* 자동 전투 구간 정산 주기
* 오프라인 보상 효율

## 7-5. 이상 탐지 포인트

* 비정상적으로 높은 골드 획득량
* 짧은 시간 내 과도한 스테이지 돌파
* 동일 세션 중복 정산
* 비정상 스탯 값
* 클라이언트 전투 표시와 서버 결과 반복 불일치

## 7-6. 고객 대응 포인트

유저 문의가 자주 발생할 수 있는 영역:

* “보스 잡았는데 클리어가 안 됨”
* “오프라인 보상이 적음”
* “전투 중 종료했는데 진행이 초기화됨”
* “프리셋 바꿨는데 바로 반영 안 됨”

이를 위해 운영툴에서 아래 조회가 가능해야 한다.

* 마지막 전투 스냅샷
* 마지막 정산 시간
* 마지막 보스 결과
* 오프라인 보상 계산 근거
* 최근 10회 전투 로그
