의도적으로 복잡한 조합 시스템은 빼고, **“장착 / 순서 / 프리셋 / 적용 타이밍”** 중심으로 단순하게 설계했어.

# 스킬 세팅 시스템 기획서

## 전제

* 본 문서는 **#15 스킬(획득/강화)** 에서 이미 획득한 스킬을, **전투에 장착하고 프리셋으로 관리하는 기능(#21)** 을 정의한다.
* 간소화 원칙에 따라 다음을 기준으로 설계한다.

  * 전투 장착 슬롯: **4개**
  * 프리셋: **3개**
  * 동일 프리셋 내 동일 스킬 중복 장착: **불가**
  * 슬롯 번호는 **자동 전투 시 우선 사용 순서**를 의미
  * 일반 필드에서는 **다음 웨이브부터 반영**, 던전 진행 중에는 **즉시 변경 불가**

---

## 1. 시스템 개요

### 1-1. 목적

스킬 세팅 시스템은 유저가 획득한 스킬 중 실제 전투에 사용할 스킬을 선택하고,
상황별로 프리셋을 저장하여 **사냥 / 보스 / 던전 대응**을 쉽게 하도록 돕는 시스템이다.

### 1-2. 시스템 역할

* 획득한 스킬 중 전투 장착 스킬 선택
* 장착 순서 설정
* 프리셋 저장 및 전환
* 현재 활성 프리셋 관리
* 오프라인 보상/전투 계산에 사용될 기준 세팅 제공

### 1-3. 설계 의도

* 방치형 게임 특성상 조작 부담을 줄이기 위해 **복잡한 수동 입력 없이 빠르게 세팅 가능**해야 한다.
* 강화는 #15에서 처리하고, #21에서는 **“어떤 스킬을 어떤 순서로 쓸지”**만 담당한다.
* 스킬 세팅 변경이 보상 계산과 연결되므로, **서버 기준의 유효성 검증**이 필요하다.

### 1-4. 핵심 정책

* 전투 장착 슬롯은 4개만 제공한다.
* 슬롯 순서는 자동 전투에서의 **발동 우선순위**를 의미한다.
* 프리셋은 3개까지 제공한다.
* 프리셋 이름 변경 기능은 초기 버전에서는 제공하지 않는다.
* 전투 밸런스 악용 방지를 위해 **던전 진행 중 프리셋 변경은 허용하지 않는다**.

---

## 2. 규칙 설명

### 2-1. 시스템 오픈 조건

* 최초로 전투 스킬을 1개 이상 획득하면 스킬 세팅 메뉴가 오픈된다.
* 프리셋 해금 조건

  * 프리셋 1: 기본 오픈
  * 프리셋 2: 지역 10 클리어
  * 프리셋 3: 지역 25 클리어

### 2-2. 장착 규칙

* 장착 가능한 대상은 **#15에서 획득한 전투용 스킬**만이다.
* 총 4개의 슬롯에 스킬을 배치할 수 있다.
* 하나의 프리셋 안에서는 **같은 스킬을 두 번 장착할 수 없다**.
* 빈 슬롯 상태로 저장은 가능하지만, 저장 시 경고를 출력한다.
* 스킬의 레벨/등급/강화 수치는 #15 데이터를 그대로 참조한다.
* 스킬 세팅에서는 스킬 성장을 변경할 수 없고, 장착 상태만 변경할 수 있다.

### 2-3. 슬롯 순서 규칙

* 슬롯 1 → 슬롯 2 → 슬롯 3 → 슬롯 4 순서로 우선 발동 판정을 한다.
* 여러 스킬의 발동 조건이 동시에 충족되면 **더 앞 슬롯의 스킬이 먼저 사용**된다.
* 모든 스킬이 쿨타임이거나 조건 미충족이면 기본 공격 또는 일반 전투 로직을 수행한다.

### 2-4. 프리셋 규칙

* 유저는 최대 3개의 프리셋을 저장할 수 있다.
* 프리셋은 각각 독립된 장착 구성을 가진다.
* 프리셋 저장과 활성 프리셋 전환은 별도 개념이다.

  * 저장: 해당 프리셋 구성을 저장
  * 적용: 저장된 프리셋 중 현재 전투에 사용할 프리셋 지정
* 프리셋 전환에는 재화를 소모하지 않는다.

### 2-5. 적용 시점 규칙

* **일반 필드 전투**: 저장/적용 완료 후 **다음 웨이브부터 반영**
* **던전/보스 전투**: 입장 후에는 프리셋 변경 불가, **다음 입장부터 반영**
* **오프라인 보상**: 마지막으로 서버에 저장된 활성 프리셋 기준으로 계산
* **랭킹/전투력 표시**: 저장 완료 시점의 활성 프리셋 기준으로 갱신

### 2-6. UX 보조 규칙

* 저장하지 않은 변경이 있으면 프리셋 탭에 **미저장 표시**를 노출한다.
* 다른 프리셋으로 이동하거나 화면을 닫을 때, 미저장 상태면 저장 여부를 묻는다.
* 장착된 스킬은 보유 스킬 목록에서 “장착 중” 표기를 한다.

## 4. UI 구성

### 4-1. 진입 경로

* 메인 화면 → 스킬 메뉴 → **세팅 탭**
* 또는 메인 HUD에 “스킬 세팅” 바로가기 버튼 제공 가능

### 4-2. 화면 구성

#### 상단 영역

* 화면 제목: 스킬 세팅
* 프리셋 탭: 프리셋 1 / 프리셋 2 / 프리셋 3
* 현재 활성 프리셋 표시
* 잠금 프리셋은 자물쇠 아이콘과 해금 조건 표시

#### 중앙 영역

* 장착 슬롯 4개 표시
* 슬롯 번호 명시: 1, 2, 3, 4
* 슬롯 번호 옆에 “우선순위” 의미를 작게 안내
* 장착된 스킬 아이콘, 희귀도 프레임, 레벨 표시

#### 하단 영역

* 보유 스킬 리스트
* 정렬/필터

  * 희귀도순
  * 레벨순
  * 역할 태그순
* 선택한 스킬 상세 패널

  * 스킬명
  * 현재 레벨
  * 쿨타임
  * 핵심 효과 요약
  * 장착/해제 버튼

### 4-3. 주요 버튼

* **저장**: 현재 프리셋 구성 저장
* **적용**: 현재 선택 프리셋을 활성 프리셋으로 지정
* **해제**: 선택한 슬롯의 스킬 제거
* **모두 해제**: 현재 프리셋 전체 슬롯 비우기
* **닫기**: 화면 종료

### 4-4. 인터랙션 방식

* 기본 방식은 **탭 장착**으로 한다.

  * 보유 스킬 선택 → 장착할 슬롯 선택
* 드래그 앤 드롭은 있으면 좋지만 초기 버전에서는 필수가 아니다.
* 장착된 스킬을 다른 슬롯에 넣으면 기존 슬롯과 교체 또는 이동 처리한다.

### 4-5. 시각 피드백

* 현재 활성 프리셋: 강조 테두리
* 미저장 프리셋: 붉은 점 또는 “저장 필요” 표시
* 빈 슬롯: 플러스 아이콘 또는 “비어 있음”
* 장착 중 스킬: 보유 리스트에서 배지 표시

## 5. 예외 처리

### 5-1. 장착 관련 예외

1. **보유하지 않은 스킬 장착 시도**

   * 클라이언트에서 우선 차단
   * 서버에서도 재검증 후 실패 응답 반환
   * 에러 문구: “보유하지 않은 스킬입니다.”

2. **동일 프리셋 내 중복 장착 시도**

   * 저장 불가
   * 에러 문구: “동일한 스킬은 한 프리셋에 중복 장착할 수 없습니다.”

3. **잠긴 프리셋 선택 시도**

   * 탭 선택 불가
   * 해금 조건 토스트 출력

4. **빈 슬롯 포함 상태 저장**

   * 저장 전 확인 팝업 출력
   * 유저가 동의하면 저장 허용

---

### 5-2. 상태 전환 관련 예외

5. **미저장 상태에서 화면 이탈**

   * 팝업 노출
   * 선택지: 저장 후 나가기 / 저장하지 않고 나가기 / 취소

6. **던전 진행 중 프리셋 적용 시도**

   * 적용 불가
   * 에러 문구: “던전 진행 중에는 프리셋을 변경할 수 없습니다.”

7. **네트워크 실패 중 저장**

   * 서버 저장 실패 시 클라이언트는 로컬 편집 상태 유지
   * 유저에게 재시도 안내
   * 실제 전투 반영은 하지 않음

---

### 5-3. 데이터 불일치 예외

8. **패치 후 삭제/비활성화된 스킬이 프리셋에 포함된 경우**

   * 서버 로드 시 해당 슬롯을 자동으로 비움
   * 클라이언트 진입 시 “사용할 수 없는 스킬이 해제되었습니다” 안내

9. **활성 프리셋 데이터 손상**

   * 서버는 가장 최근 정상 저장된 프리셋으로 롤백
   * 롤백 가능한 데이터가 없으면 프리셋 1 기준 빈 슬롯 상태로 복구

10. **동시 저장 충돌**

* `version` 값 불일치 시 저장 실패
* 최신 데이터를 다시 받아오고 재편집하도록 유도

## 6. 서버/클라이언트 처리 구분

### 6-1. 클라이언트 처리

* 프리셋 UI 렌더링
* 보유 스킬 리스트 정렬/필터
* 슬롯 선택/장착/해제 인터랙션
* 미저장 상태 관리
* 저장 전 전투력 미리보기 계산
* 에러/경고 팝업 출력

### 6-2. 서버 처리

* 보유 여부 검증
* 프리셋 잠금 여부 검증
* 동일 스킬 중복 여부 검증
* 저장/적용 요청 처리
* 활성 프리셋 저장
* 오프라인 보상 계산용 세팅 반영
* 랭킹/전투력 산정용 파생값 갱신
* 비정상 데이터 정리 및 복구

### 6-3. 처리 원칙

* **UI 편집은 클라이언트**
* **유효성 검증과 최종 저장은 서버**
* **보상/오프라인 계산 기준은 서버 저장값**
* 클라이언트는 연출상 즉시 반영해 보여줄 수 있으나, 실제 보상 반영 기준은 서버값이다.

### 6-4. 전투 반영 기준

* 필드 전투 연출은 클라이언트에서 자연스럽게 갱신 가능
* 오프라인 보상, 던전 결과, 랭킹 반영 등 가치 있는 결과는 서버 기준
* 즉, **“보이는 전투”와 “가치가 있는 결과”는 분리**한다.

### 7-2. 운영 지표

1. **스킬 장착률**

   * 획득한 스킬 중 실제로 장착되는 비율
   * 특정 스킬이 거의 장착되지 않으면 밸런스/가독성 점검 필요

2. **슬롯별 사용률**

   * 슬롯 1~4에 어떤 스킬이 자주 배치되는지 확인
   * 특정 슬롯 편중이 심하면 우선순위 규칙 재검토 가능

3. **프리셋 활용률**

   * 프리셋 2, 3의 사용률
   * 사용률이 너무 낮으면 해금 시점이 늦거나 UI 가시성이 부족한 것

4. **저장 실패율**

   * 네트워크 실패
   * 버전 충돌
   * 검증 실패
   * 저장 실패율이 높으면 UX 개선 필요

5. **획득 후 미장착 스킬 비율**

   * 새로 얻은 스킬이 장착으로 이어지지 않으면 해당 스킬의 효용 체감이 낮은 것

6. **콘텐츠별 프리셋 전환 빈도**

   * 던전 입장 전 프리셋 전환이 활발한지 확인
   * 전환이 거의 없으면 프리셋 필요성이 약하다는 의미

---

### 7-3. 운영 체크 포인트

* 특정 스킬 장착률이 전체 유저의 70% 이상으로 과도하게 높으면 OP 가능성 점검
* 특정 스킬 장착률이 5% 이하로 장기간 유지되면 구조적 저성능 가능성 점검
* 프리셋 2, 3 해금 이후 사용률이 낮으면 해금 위치 조정 검토
* 저장 실패/충돌이 잦으면 클라-서버 동기화 방식 개선 필요

# 요약 설계안

이 시스템의 핵심은 아래 5개야.

* **장착 슬롯 4개**
* **프리셋 3개**
* **슬롯 순서 = 자동 전투 우선순위**
* **필드는 다음 웨이브, 던전은 다음 입장부터 반영**
* **서버 저장값 기준으로 오프라인 보상/랭킹 계산**

원하면 다음에는 이어서 바로
**1) 스킬 세팅 UI 와이어 설명서** 또는
**2) 이 기획서를 기반으로 한 서버 API 명세서** 형태로 확장해줄게.
