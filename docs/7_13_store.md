# 상점 시스템 기획서

## 전제

본 문서는 아래 공통 설계 기준을 따르는 **간소화 방치형 RPG의 상점 시스템**을 정의한다.

* 사용 재화: 골드, 다이아, 장비 소환권, 스킬 소환권, 던전 티켓, 길드 코인
* 장비 완제품 판매 없음
* 스킬 완제품 판매 없음
* 상점은 **성장 보조 및 재화 소모처** 역할에 집중
* 시즌패스/이벤트는 별도 시스템이며, 상점은 필요 시 해당 시스템의 진입 허브 역할만 수행

---

## 1. 시스템 개요

### 1-1. 시스템 목적

상점은 유저가 현재 막힌 성장 구간을 짧게 보완할 수 있도록 돕는 시스템이다.
직접 전투력을 완성품 형태로 판매하지 않고, **소환권·티켓·골드** 중심으로 성장 재화를 공급하여 기존 코어 루프를 보조한다.

핵심 목적은 다음 4가지다.

1. **다이아/골드/길드 코인의 안정적인 소모처 제공**
2. **일일 접속 동기 부여**
3. **성장 정체 구간 완화**
4. **길드 참여 보상 체감 강화**

### 1-2. 상점 구성 범위

1차 버전에서는 상점을 아래 3개 탭으로만 구성한다.

| 탭     | 설명                   | 주요 사용 재화 |
| ----- | -------------------- | -------- |
| 일일 상점 | 매일 리셋되는 할인 상품        | 골드, 다이아  |
| 일반 상점 | 항상 노출되는 상시 상품        | 다이아      |
| 길드 상점 | 길드 가입 시 이용 가능한 전용 상품 | 길드 코인    |

### 1-3. 설계 방향

* **랜덤 진열 방식 미사용**
* **수동 새로고침 기능 미사용**
* **완제품 판매 미사용**
* **상품 수 최소화**
* **서버 시간 기준 일일 리셋**

즉, 복잡한 확률형 상점이 아니라 **고정된 상품 목록 + 구매 제한 + 일일 리셋** 구조로 설계한다.

## 2. 규칙 설명

## 2-1. 이용 조건

* 상점은 **튜토리얼 종료 후** 메인 화면에서 이용 가능
* 길드 상점은 **길드 가입 후** 해금
* 특정 탭 또는 상품은 필요 시 지역 진척도 조건을 둘 수 있으나, 1차 버전에서는 최소화

## 2-2. 리셋 규칙

* 일일 상점 및 구매 제한은 **매일 00:00 (서버 시간 기준)** 초기화
* 일반 상점은 항상 열려 있으나, 개별 상품에 일일 구매 제한을 둘 수 있음
* 길드 상점도 운영 편의상 동일하게 **일일 초기화** 적용

## 2-3. 상품 판매 원칙

상점은 아래 원칙을 따른다.

1. **장비 완제품 판매 금지**
   장비는 반드시 장비 소환권을 통해 획득하도록 유지

2. **스킬 완제품 판매 금지**
   스킬은 반드시 스킬 소환권을 통해 획득하도록 유지

3. **지역 진입 자체를 여는 상품 판매 금지**
   상점이 콘텐츠 해금 수단이 되지 않도록 함

4. **길드 랭킹 점수 직접 판매 금지**
   길드 상점은 보상 소비처이지 랭킹 조작 수단이 아님

## 2-4. 탭별 규칙

### A. 일일 상점

* 매일 리셋되는 한정 상품 제공
* 골드 소모처와 다이아 소모처를 동시에 담당
* 유저가 매일 접속해 소량의 성장 재화를 확보하도록 유도

예시 상품:

| 상품명       | 가격  | 구매 제한 | 비고       |
| --------- | --- | ----- | -------- |
| 장비 소환권 x1 | 골드  | 1일 1회 | 골드 소모처   |
| 스킬 소환권 x1 | 골드  | 1일 1회 | 골드 소모처   |
| 던전 티켓 x1  | 골드  | 1일 1회 | 골드 소모처   |
| 골드 꾸러미    | 다이아 | 1일 3회 | 정체 구간 보조 |
| 장비 소환권 x5 | 다이아 | 1일 2회 | 중기 성장 보조 |
| 스킬 소환권 x5 | 다이아 | 1일 2회 | 중기 성장 보조 |

> 수치 밸런스는 예시이며 실제 값은 경제 밸런싱 단계에서 조정

### B. 일반 상점

* 항상 같은 상품을 노출
* 다이아를 사용해 부족한 성장 자원을 구매
* 과금/무과금 모두 접근 가능하되 과도한 우회 성장을 막기 위해 구매 제한 적용

예시 상품:

| 상품명        | 가격  | 구매 제한 | 비고     |
| ---------- | --- | ----- | ------ |
| 골드 꾸러미(소)  | 다이아 | 1일 5회 | 초반 보조  |
| 장비 소환권 x10 | 다이아 | 1일 3회 | 핵심 상품  |
| 스킬 소환권 x10 | 다이아 | 1일 3회 | 핵심 상품  |
| 던전 티켓 x3   | 다이아 | 1일 5회 | 콘텐츠 보조 |

### C. 길드 상점

* 길드 코인 소비처
* 길드 콘텐츠 참여의 동기 제공
* 핵심 성장 재화를 보조하되, 일반 상점보다 효율이 좋거나 유사하도록 설계 가능

예시 상품:

| 상품명       | 가격    | 구매 제한 | 비고          |
| --------- | ----- | ----- | ----------- |
| 장비 소환권 x3 | 길드 코인 | 1일 2회 | 길드 활동 보상 체감 |
| 스킬 소환권 x3 | 길드 코인 | 1일 2회 | 길드 활동 보상 체감 |
| 던전 티켓 x2  | 길드 코인 | 1일 2회 | 반복 플레이 유도   |
| 골드 꾸러미    | 길드 코인 | 1일 2회 | 보조 소모처      |

## 2-5. 구매 처리 규칙

구매는 아래 순서로 처리한다.

1. 유저가 상품 선택
2. 구매 확인 팝업 노출
3. 서버에 구매 요청 전송
4. 서버가 상품 유효성/구매 제한/재화 보유량 검증
5. 서버가 재화 차감 및 보상 지급
6. 서버가 구매 이력 갱신
7. 클라이언트가 최신 재화와 남은 구매 횟수 반영

## 2-6. 지급 규칙

* 구매 보상은 **즉시 지급**
* 지급 대상은 재화형이므로 별도 장비 인벤토리 부족 이슈 없음
* 지급 실패 시 **서버 롤백 우선**
* 롤백 불가 예외 상황에서는 **우편 보상 복구** 사용 가능

## 4. UI 구성

## 4-1. 화면 구조

모바일 세로 화면 기준으로 아래와 같이 구성한다.

### 상단 영역

* 화면 타이틀: 상점
* 재화 바: 골드 / 다이아 / 장비 소환권 / 스킬 소환권 / 던전 티켓 / 길드 코인
* 닫기 버튼

### 탭 영역

* 일일 상점
* 일반 상점
* 길드 상점

### 본문 영역

상품 카드 리스트를 세로 스크롤로 배치한다.

상품 카드에는 아래 요소가 포함된다.

* 상품 아이콘
* 상품명
* 보상 수량
* 가격 아이콘 + 수량
* 남은 구매 횟수
* 할인/한정 태그
* 구매 버튼

### 하단/보조 영역

* 일일 리셋 남은 시간 표시
* 길드 미가입 시 길드 상점 잠금 안내 문구
* 부족 재화 시 획득처 안내 버튼

## 4-2. 구매 팝업

상품 클릭 시 확인 팝업을 노출한다.

표시 요소:

* 상품명
* 획득 보상
* 소모 재화
* 현재 보유 재화
* 구매 후 예상 잔액
* 확인 / 취소 버튼

## 4-3. UI 정책

* 품절 상품은 카드 유지 + `구매 완료` 또는 `품절` 상태 표시
* 구매 불가 시 버튼 비활성화
* 잠긴 길드 상점은 탭 노출 가능하나 오버레이로 잠금 표시
* 서버 기준 리셋 타이머 사용
* 가격/보상은 서버 응답값 기준으로 렌더링

## 5. 예외 처리

## 5-1. 재화 부족

상황:

* 보유 재화가 가격보다 적음

처리:

* 구매 요청 전 클라이언트 1차 체크
* 서버에서 최종 검증
* 실패 시 `재화가 부족합니다` 메시지 노출
* 가능하면 해당 재화 획득처 이동 버튼 제공

## 5-2. 구매 제한 초과

상황:

* 일일 구매 횟수 초과

처리:

* 서버에서 최종 거부
* 남은 횟수 0으로 갱신
* 카드 상태를 즉시 품절 처리

## 5-3. 상품 판매 종료

상황:

* 상점을 열어둔 상태에서 운영자가 상품 비활성화 또는 기간 종료

처리:

* 구매 시 서버가 거부
* `판매가 종료된 상품입니다` 메시지 출력
* 상점 목록 재요청

## 5-4. 중복 구매 요청

상황:

* 유저가 빠르게 연타하거나 네트워크 지연으로 중복 요청 발생

처리:

* 서버에서 요청 단위 idempotency 처리 권장
* 동일 요청 중복 수신 시 1회만 처리
* 클라이언트는 구매 요청 중 버튼 잠금

## 5-5. 구매 성공 후 응답 누락

상황:

* 서버는 처리 완료했지만 클라이언트가 결과를 못 받음

처리:

* 재접속 또는 상점 재진입 시 서버 데이터 기준으로 재동기화
* 중복 구매 방지 위해 클라이언트 재시도 전에 상태 조회
* 필요 시 우편/로그 기반 CS 복구 가능

## 5-6. 길드 상태 변화

상황:

* 길드 상점 이용 중 추방/탈퇴 발생

처리:

* 구매 시점에 길드 소속 여부를 서버 재검증
* 미소속이면 구매 실패 및 길드 탭 잠금 갱신

## 5-7. 가격 변경

상황:

* 운영 중 상품 가격이 변경됨

처리:

* 클라이언트가 오래된 가격으로 요청해도 서버 가격 기준 적용 불가
* 서버는 `상품 정보가 갱신되었습니다` 오류 반환
* 카탈로그 재호출 후 최신 정보 반영

## 6. 서버/클라이언트 처리 구분

## 6-1. 서버 담당

서버는 상점의 **권한 주체**다.

### 서버 처리 항목

* 상점 상품 목록 제공
* 상품 노출 가능 여부 판정
* 길드 가입 여부 판정
* 판매 기간 확인
* 구매 횟수 확인
* 재화 보유량 검증
* 재화 차감
* 보상 지급
* 구매 로그 저장
* 리셋 기준 시간 제공

즉, **가격, 제한, 지급, 차감은 모두 서버 authoritative**로 처리한다.

## 6-2. 클라이언트 담당

클라이언트는 상점의 **표시 및 입력 인터페이스**를 담당한다.

### 클라이언트 처리 항목

* 탭 UI 렌더링
* 상품 카드 표시
* 리셋 타이머 표시
* 구매 팝업 노출
* 부족 재화 시 시각적 안내
* 구매 요청 전송
* 성공/실패 결과 반영

## 6-3. 권장 구매 플로우

```text
상점 진입
→ 서버에서 카탈로그 수신
→ 유저가 상품 선택
→ 클라이언트 팝업 노출
→ 구매 요청 전송
→ 서버 검증
→ 서버 차감/지급/로그 기록
→ 결과 응답
→ 클라이언트 재화/남은 횟수 갱신
```

## 7-2. 운영 포인트

### A. 경제 밸런스 점검

운영에서는 아래 수치를 주기적으로 본다.

* 상점 진입률
* 탭별 진입 비중
* 상품별 구매율
* 상품별 구매 실패율
* 다이아 소모량
* 골드 소모량
* 길드 코인 소모량
* 특정 성장 구간에서의 구매 집중도

### B. 정체 구간 파악

예를 들어 특정 지역에서 이탈률이 높고 동시에 골드 꾸러미/소환권 구매율이 급증하면, 해당 구간 난이도가 과한 신호일 수 있다.

즉 상점 로그는 단순 매출 지표가 아니라 **밸런스 문제 탐지 도구**로 활용한다.

### C. 운영 조정 항목

라이브 운영에서는 아래 값들을 마스터 데이터로 조정 가능하게 하는 것이 좋다.

* 가격
* 구매 제한
* 판매 기간
* 노출 여부
* 상품 정렬 순서
* 특정 조건 해금 여부

### D. 악용 탐지

아래 경우는 이상 로그로 분류한다.

* 짧은 시간 내 비정상 연속 구매 시도
* 실패 로그 급증
* 특정 상품만 과도하게 구매
* 길드 탈퇴/가입 반복과 길드 상점 구매 패턴 연동

# 요약 설계안

이 상점은 복잡한 확률 상점이 아니라,

* **일일 상점**으로 접속 동기를 만들고,
* **일반 상점**으로 성장 보조 재화를 제공하며,
* **길드 상점**으로 길드 활동 보상을 소비하게 하는

아주 단순한 구조다.

핵심은 **완제품을 직접 팔지 않고**,
**소환권/티켓/골드 중심의 보조 성장만 허용하는 것**이다.

원하면 다음으로 이어서 **이 문서 스타일 그대로 `퀘스트` 또는 `소환` 시스템 기획서**도 같은 형식으로 맞춰서 작성해줄게.
