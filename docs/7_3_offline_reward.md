좋아. 아래는 네 기준에 맞춘 **#9 오프라인 보상 정보 시스템 기획서** 초안이야.
이번 문서는 **“임모탈 키우기보다 단순한 구조”**를 기준으로, **오프라인 보상은 골드 + 장비 드랍 중심**으로 설계했어.
즉, **다이아 / 장비 소환권 / 스킬 소환권 / 던전 티켓 / 길드 코인**은 오프라인 보상에서 제외하는 권장안이야. 그래야 경제 구조가 깔끔해지고 악용 포인트도 줄어.

# 9. 오프라인 보상 정보 시스템 기획서

## 1. 시스템 개요

### 1-1. 시스템 목적

오프라인 보상 정보 시스템은 플레이어가 게임을 종료한 동안에도 일정 수준의 성장을 유지할 수 있도록 지원하는 시스템이다.
방치형 게임의 핵심 만족감인 “접속하지 않아도 조금씩 성장한다”는 경험을 제공하며, 복귀 시 즉시 보상을 확인하고 수령하는 흐름을 만든다.

### 1-2. 시스템 역할

이 시스템은 전체 코어 루프 내에서 다음 역할을 담당한다.

* 접속 공백에 대한 손실 체감 완화
* 복귀 즉시 성장 체감 제공
* 지역 진입 → 사냥 → 강화 → 재진입 루프의 이탈 구간 보완
* 장시간 미접속 유저의 복귀 동기 제공

### 1-3. 설계 원칙

* **단순성 우선**: 복잡한 추가 재화 없이 이해하기 쉬워야 한다.
* **예측 가능성**: 유저가 “얼마를 받을지” 대략 예측 가능해야 한다.
* **악용 방지**: 시간 조작, 중복 수령, 특정 화면 종료 악용 등을 막아야 한다.
* **서버 기준 처리**: 시간 및 보상 계산은 서버 기준으로 처리한다.
* **12시간 상한**: 최대 누적 시간은 12시간으로 제한한다.

### 1-4. 초기 버전 지급 항목

초기 버전에서는 오프라인 보상을 아래 2종으로 제한한다.

* **골드**
* **장비 드랍** (무기 / 방어구 / 반지 / 목걸이)

### 1-5. 초기 버전 제외 항목

아래 항목은 오프라인 보상에서 제외한다.

* 다이아
* 장비 소환권
* 스킬 소환권
* 던전 티켓
* 길드 코인

### 1-6. 해금 조건

오프라인 보상 시스템은 **지역 선택 기능이 해금된 이후**부터 활성화된다.
튜토리얼 단계에서는 비활성화한다.

## 2. 규칙 설명

## 2-1. 기본 규칙

* 유저가 게임을 종료하거나 접속이 끊기면 오프라인 시간이 누적된다.
* 다음 로그인 시 누적된 시간에 비례해 보상이 생성된다.
* 최대 누적 시간은 **12시간(43,200초)** 이다.
* 오프라인 상태에서는 **지역 자동 돌파는 발생하지 않는다.**
* 오프라인 보상은 **마지막 일반 지역 사냥 기준**으로 계산된다.

## 2-2. 사냥 기준 지역

오프라인 보상 계산에 사용하는 기준 지역은 아래 규칙을 따른다.

1. 마지막으로 진입해 있던 **일반 지역**을 기준으로 한다.
2. 로그아웃 당시 던전/길드/메뉴 화면에 있었다면, **마지막 일반 지역**을 사용한다.
3. 보스 연출, 로딩, 일시적인 전투 중단 등 특수 상태에서는 **직전 확정 일반 지역**을 사용한다.

즉, 오프라인 보상은 “현재 사냥 중인 일반 지역” 기준으로만 계산된다.

## 2-3. 오프라인 시간 계산

```text
offline_sec = min(server_login_time - server_logout_time, 43200)
```

* 43,200초를 초과하는 시간은 버린다.
* 클라이언트 시간은 사용하지 않는다.
* 강제 종료 / 앱 크래시도 서버 마지막 접속 시점을 기준으로 계산한다.

## 2-4. 최소 인정 시간

* 오프라인 시간이 **60초 미만**이면 보상을 지급하지 않는다.
* 너무 짧은 재접속에서 팝업이 반복 노출되는 것을 방지하기 위함이다.

## 2-5. 보상 계산 방식

초기 버전은 **“실제 전투 시뮬레이션”이 아니라 “지역별 고정 보상 테이블”**을 사용한다.
즉, 접속 종료 당시 DPS를 복잡하게 다시 계산하지 않고, 지역별 시간당 보상량으로 단순 계산한다.

### 골드 계산식

```text
gold_reward = floor(region_gold_per_sec × offline_sec)
```

### 장비 드랍 횟수 계산식

```text
equipment_roll_count = floor(region_equipment_roll_per_sec × offline_sec)
```

장비는 `equipment_roll_count` 만큼 드랍 테이블을 굴려 생성한다.

### 장비 생성 규칙

* 드랍 가능한 부위는 4종만 존재한다.

  * 무기
  * 방어구
  * 반지
  * 목걸이
* 장비 등급/옵션은 해당 지역의 장비 드랍 테이블을 따른다.
* 오프라인 보상은 **장비 획득/분해/판매의 대체 루트** 역할을 한다.

## 2-6. 오프라인 중 진행되지 않는 것

오프라인 상태에서는 아래가 진행되지 않는다.

* 지역 클리어 진척
* 보스 처치 판정
* 던전 입장/소모
* 길드 활동
* 시즌패스 미션 직접 진행
* 이벤트 수동 참여 판정

즉, **오프라인은 일반 지역의 정적 파밍만 수행**하는 구조다.

## 2-7. 수령 규칙

* 로그인 시 미수령 오프라인 보상이 존재하면 팝업을 노출한다.
* 유저는 즉시 수령하거나 닫을 수 있다.
* 팝업을 닫아도 보상은 사라지지 않는다.
* 미수령 보상은 **#9 오프라인 보상 정보 화면**에서 다시 확인하고 수령할 수 있다.

## 2-8. 미수령 보상 누적 규칙

* 유저가 보상을 수령하지 않은 채 다시 종료하더라도, 기존 미수령 보상은 유지된다.
* 다음 로그인 시 **기존 미수령 보상 + 신규 오프라인 보상**을 합산하여 보여준다.

이 규칙을 적용하면 유저가 팝업을 실수로 닫아도 손해를 보지 않는다.

## 2-9. 권장 운영 규칙

* **오프라인 보상은 직접 플레이보다 효율이 낮아야 한다.**
* 권장 기준:

  * 골드는 직접 플레이 대비 60~80% 수준
  * 장비 드랍도 직접 플레이 대비 낮은 기대값
* 이유는 접속 플레이의 가치를 유지하기 위함이다.

## 4. UI 구성

## 4-1. 진입 위치

1. **로그인 직후 팝업 자동 노출**
2. 메인 화면의 **#9 오프라인 보상 정보 버튼**
3. 미수령 보상이 있을 경우 해당 버튼에 **레드닷 표시**

## 4-2. 로그인 팝업 구성

### 상단 영역

* 제목: `오프라인 보상`
* 부제: `자리를 비운 동안 획득한 보상입니다`
* 누적 시간 표기: `6시간 12분`
* 최대 시간 안내: `최대 12시간까지 누적`

### 중앙 정보 영역

* 기준 지역명
* 획득 골드 총량
* 획득 장비 개수
* 장비 미리보기(대표 몇 개 + 나머지 개수 표기)

예시:

* 골드 180,000
* 장비 7개 획득
* 희귀 장비 1개 포함

### 하단 버튼 영역

* `수령`
* `닫기`

`닫기`를 눌러도 보상은 유지된다.

## 4-3. #9 오프라인 보상 정보 화면 구성

이 화면은 단순 팝업이 아니라 **정보성 화면** 역할도 해야 한다.

표시 항목:

* 현재 미수령 보상 여부
* 마지막 기준 지역
* 현재 오프라인 누적 가능 최대 시간: 12시간
* 해당 지역의 시간당 예상 획득량

  * 시간당 골드
  * 시간당 장비 획득 수
* 현재 미수령 보상 상세 목록
* 수령 버튼

## 4-4. 보상 상세 표시 방식

장비가 많을 경우 한 화면에 전부 나열하지 않고 아래 방식 사용 권장:

* 상단: 요약

  * 총 골드
  * 총 장비 수
  * 희귀 이상 장비 수
* 하단: 상세 리스트

  * 최근 획득순 또는 등급 높은 순 정렬

## 4-5. 인벤토리 초과 UI

인벤토리 부족으로 일부 장비가 자동 판매 처리된 경우, 아래 문구를 별도로 노출한다.

예시:

* `인벤토리가 가득 차 일부 장비가 자동 판매되었습니다`
* `자동 판매 골드: 12,500`

## 5. 예외 처리

## 5-1. 짧은 재접속

* 60초 미만 오프라인은 보상 미지급
* 팝업 미노출

## 5-2. 인벤토리 부족

장비 인벤토리가 가득 찬 경우:

* 초과 장비는 자동 판매 처리
* 판매 금액은 골드로 환산
* 환산된 골드는 오프라인 보상에 합산
* UI에 별도 표시

이 방식이 가장 단순하고 운영 부담이 적다.

## 5-3. 중복 수령 요청

* 유저가 수령 버튼을 연속으로 누르거나 재전송해도 1회만 지급되어야 한다.
* 서버는 **보상 ID 기준 idempotent 처리**를 한다.
* 이미 수령된 요청은 실패가 아니라 “중복 요청 무시”로 처리 가능하다.

## 5-4. 시간 조작

* 로컬 기기 시간은 절대 사용하지 않는다.
* 서버 로그인/로그아웃 시각 기준으로만 계산한다.

## 5-5. 비정상 종료

* 앱 크래시, 강제 종료, 네트워크 끊김 발생 시에도 서버 마지막 세션 종료 시각 기준으로 계산한다.
* 단, 순간적인 연결 불안정 대응을 위해 **30초 유예 시간**을 둘 수 있다.

권장:

* 30초 이내 재연결 시 동일 세션으로 간주
* 30초 초과 시 오프라인 시작 처리

## 5-6. 던전/특수 콘텐츠 종료 상태

로그아웃 시 던전, 길드 화면, 설정, 팝업 상태였더라도:

* 오프라인 보상은 던전 기준으로 계산하지 않음
* 마지막 일반 지역 기준으로만 계산

## 5-7. 데이터 버전 변경

패치나 밸런스 변경으로 지역 보상량이 바뀌는 경우:

* 보상 계산 시 사용한 `reward_version`을 저장
* 이미 생성된 미수령 보상은 생성 당시 버전 기준 유지
* 그래야 로그인 후 보상이 바뀌는 문제를 막을 수 있다

## 5-8. 보상 지급 실패

보상 생성은 되었으나 지급 도중 장애가 발생한 경우:

* 서버는 지급 완료 전까지 `미수령 상태`를 유지
* 지급 성공 시에만 `수령 완료`로 변경
* 부분 지급은 지양하고, 원자적으로 처리하는 것이 좋다

## 5-9. 계정 다중 접속

동일 계정이 여러 기기에서 접속한 경우:

* 마지막 정상 세션 기준으로 오프라인 계산
* 동시에 여러 기기에서 같은 보상을 받지 못하게 한다

---

## 6. 서버/클라이언트 처리 구분

## 6-1. 서버 처리

서버가 담당해야 하는 항목:

* 마지막 로그인/로그아웃 시각 저장
* 마지막 일반 지역 저장
* 오프라인 시간 계산
* 지역 보상 테이블 조회
* 골드/장비 보상 생성
* 미수령 보상 저장
* 보상 수령 검증
* 중복 수령 방지
* 로그 저장
* 인벤토리 초과 처리
* 보상 지급 원자성 보장

즉, **보상 계산과 지급은 전부 서버 권한**이다.

## 6-2. 클라이언트 처리

클라이언트가 담당해야 하는 항목:

* 로그인 시 오프라인 보상 팝업 표시
* #9 화면에서 상세 정보 렌더링
* 수령 버튼 입력 처리
* 서버 응답 결과에 따른 연출
* 레드닷 표시
* 장비 목록 및 요약 정보 정렬/표시

즉, **클라이언트는 표시와 입력만 담당**한다.

## 6-3. 권장 처리 흐름

### 로그인 시

1. 클라이언트 로그인 요청
2. 서버가 `last_logout_at` 확인
3. 서버가 오프라인 시간 계산
4. 서버가 보상 생성 후 `pending_reward_json` 저장
5. 서버가 클라이언트에 미수령 보상 정보 응답
6. 클라이언트가 오프라인 보상 팝업 노출

### 수령 시

1. 클라이언트가 수령 요청 전송
2. 서버가 미수령 보상 존재 여부 확인
3. 서버가 중복 수령 여부 확인
4. 서버가 골드/장비 지급
5. 서버가 로그 기록
6. 서버가 미수령 상태 제거
7. 클라이언트가 수령 연출 및 수치 갱신

## 6-4. 권장 API 예시

* `GET /offline-reward`

  * 현재 미수령 보상 조회
* `POST /offline-reward/claim`

  * 미수령 보상 수령
