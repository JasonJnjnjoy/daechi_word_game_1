좋아. 아래는 네가 바로 **시스템 기획서**에 넣을 수 있도록 정리한 **‘우편’ 시스템 기획안**이다.
기준은 네가 정한 **간단한 방치형 구조**에 맞춰서, 운영 보상 수령 허브 역할에 집중하도록 설계했다.

---

# 우편 시스템 기획서

## 1. 시스템 개요

### 1-1. 시스템 목적

우편 시스템은 유저에게 즉시 지급되지 않는 보상이나 운영성 보상을 **안전하게 전달하고 수령시키기 위한 시스템**이다.
주요 역할은 다음과 같다.

* 운영 보상 지급
* 점검 보상 및 오류 보상 지급
* 이벤트 보상 지급
* 시즌 패스 보상 지급
* 랭킹 보상 지급
* 길드 보상 지급
* 공지성 메시지 전달

### 1-2. 시스템 성격

본 게임의 우편은 **유저 간 송수신 기능이 없는 일방향 시스템 우편**이다.
즉, 유저가 다른 유저에게 우편을 보내는 기능은 없으며, 서버 또는 운영툴을 통해 생성된 우편만 수신 가능하다.

### 1-3. 설계 방향

간단한 방치형 게임 구조에 맞춰 우편은 다음 원칙으로 설계한다.

* **보상 수령 기능 중심**
* **복잡한 분류 최소화**
* **대량 지급과 일괄 수령 지원**
* **중복 수령 방지 및 서버 권한 처리**
* **재화 손실 방지 우선**

### 1-4. 우편에서 다루는 보상 범위

현재 우편 첨부 보상은 아래 범위만 허용한다.

* 골드
* 다이아
* 장비 소환권
* 스킬 소환권
* 던전 티켓
* 길드 코인

즉, 본 기획 범위에서는 **직접 장비 아이템이나 직접 스킬 데이터를 첨부하지 않고**,
모든 보상은 **재화/티켓 형태로만 지급**한다.

## 2. 규칙 설명

### 2-1. 기본 규칙

* 우편은 서버에서 생성되어 계정 단위로 지급된다.
* 유저는 우편함에서 개별 수령 또는 전체 수령을 할 수 있다.
* 우편은 생성 시점부터 **유효기간**을 가진다.
* 유효기간이 지난 우편은 자동 만료되며 보상을 받을 수 없다.
* 보상이 없는 공지 우편도 생성 가능하다.

### 2-2. 우편 종류

우편은 기능 기준으로 아래 3종으로 구분한다.

#### 1) 일반 공지 우편

* 첨부 보상 없음
* 운영 공지, 안내 메시지 전달 목적

#### 2) 보상 우편

* 첨부 보상 있음
* 이벤트, 시즌패스, 랭킹, 길드, 점검 보상 등에 사용

#### 3) 보상/보상성 공지 우편

* 공지 메시지 + 첨부 보상 포함
* 점검 보상, 사과 보상, 긴급 지급 등에 사용

### 2-3. 상태 규칙

우편은 아래 상태를 가진다.

* **Unread**: 읽지 않음
* **Read**: 읽음
* **Claimed**: 첨부 보상 수령 완료
* **Expired**: 유효기간 만료

### 2-4. 상태 전이

상태 흐름은 다음과 같다.

`생성 → 수신(Unread) → 열람(Read) → 보상 수령(Claimed)`

또는

`생성 → 수신/열람 상태 유지 → 유효기간 종료(Expired)`

### 2-5. 수령 규칙

* 개별 우편은 상세 팝업에서 수령 가능하다.
* 첨부 보상이 있는 우편만 수령 버튼이 노출된다.
* 전체 수령은 **수령 가능한 우편만 일괄 처리**한다.
* 이미 수령한 우편은 다시 수령할 수 없다.
* 만료된 우편은 수령할 수 없다.

### 2-6. 유효기간 규칙

* 우편은 기본적으로 **발송 시 만료일을 함께 가진다**.
* 기본 권장값은 **7일**
* 운영성 보상, 이벤트 성격에 따라 1~30일 범위 내 설정 가능
* 만료 후 우편은 보상을 잃고, 리스트에서 숨기거나 Expired 상태로 짧게 유지 후 제거한다.

### 2-7. 우편함 정리 규칙

* 우편함은 최신순 정렬
* 이미 수령 완료된 우편은 일정 기간 후 자동 삭제 가능
* 우편함 최대 저장 수를 초과할 경우, **수령 완료/만료 우편부터 자동 정리**
* 보상 미수령 우편은 자동 정리 대상에서 제외

### 2-8. 레드닷 규칙

아래 조건 중 하나라도 만족하면 우편 아이콘에 레드닷 표시

* 읽지 않은 우편이 1개 이상 존재
* 첨부 보상이 있고 아직 수령하지 않은 우편이 1개 이상 존재

## 3. 데이터 구조

우편 시스템은 크게 **우편 본문 데이터**, **첨부 보상 데이터**, **발송 관리 데이터**로 구성한다.

### 3-1. UserMail

유저가 실제로 받는 우편 데이터

| 필드명         | 타입                | 설명                                                                   |
| ----------- | ----------------- | -------------------------------------------------------------------- |
| mail_id     | bigint            | 우편 고유 ID                                                             |
| account_id  | bigint            | 수신 계정 ID                                                             |
| mail_type   | enum              | notice / reward / mixed                                              |
| source_type | enum              | system / event / season_pass / ranking / guild / maintenance / admin |
| title       | string            | 우편 제목                                                                |
| body        | text              | 우편 본문                                                                |
| is_read     | bool              | 읽음 여부                                                                |
| is_claimed  | bool              | 보상 수령 여부                                                             |
| received_at | datetime          | 수신 시각                                                                |
| expire_at   | datetime          | 만료 시각                                                                |
| read_at     | datetime nullable | 열람 시각                                                                |
| claimed_at  | datetime nullable | 수령 시각                                                                |
| batch_id    | bigint nullable   | 대량 발송 묶음 ID                                                          |
| priority    | int               | 정렬 및 강조용 우선순위                                                        |
| status      | enum              | unread / read / claimed / expired                                    |

### 3-2. MailAttachment

우편 첨부 보상 데이터

| 필드명           | 타입     | 설명                                                                         |
| ------------- | ------ | -------------------------------------------------------------------------- |
| attachment_id | bigint | 첨부 고유 ID                                                                   |
| mail_id       | bigint | 연결된 우편 ID                                                                  |
| reward_type   | enum   | currency / ticket                                                          |
| reward_key    | enum   | gold / diamond / equip_ticket / skill_ticket / dungeon_ticket / guild_coin |
| amount        | bigint | 지급 수량                                                                      |

### 3-3. MailBatch

운영툴/서버 스케줄러 기준 대량 발송 관리 데이터

| 필드명          | 타입       | 설명                                                         |
| ------------ | -------- | ---------------------------------------------------------- |
| batch_id     | bigint   | 발송 배치 ID                                                   |
| source_type  | enum     | 발송 출처                                                      |
| target_type  | enum     | all / segment / account_list / ranking_group / guild_group |
| target_desc  | string   | 발송 대상 설명                                                   |
| title        | string   | 우편 제목                                                      |
| body         | text     | 우편 본문                                                      |
| expire_at    | datetime | 만료 시각                                                      |
| created_by   | string   | 생성 주체(시스템/운영자)                                             |
| created_at   | datetime | 생성 시각                                                      |
| sent_count   | int      | 발송 수                                                       |
| failed_count | int      | 실패 수                                                       |

### 3-4. 추천 보상 구조 규칙

우편 첨부는 배열 구조로 저장한다.

예시:

```json
[
  { "reward_type": "currency", "reward_key": "gold", "amount": 50000 },
  { "reward_type": "ticket", "reward_key": "equip_ticket", "amount": 10 }
]
```

### 3-5. 데이터 설계 원칙

* 첨부 보상은 **범용 reward 구조**로 통일
* 우편 본문과 보상 데이터 분리
* 수령 시점에 **서버 트랜잭션으로 계정 재화 반영**
* 같은 우편의 중복 수령을 막기 위해 `is_claimed`와 `claimed_at`를 반드시 관리

## 4. UI 구성

## 4-1. 진입 위치

* 메인 화면 상단 또는 사이드 메뉴에 **우편 버튼** 배치
* 수령 가능/미확인 우편 존재 시 레드닷 표시

## 4-2. 우편함 메인 화면 구성

### 상단 영역

* 화면명: 우편
* 전체 수령 버튼
* 닫기 버튼

### 리스트 영역

각 우편 셀에는 아래 정보 노출

* 제목
* 발송 출처 아이콘 또는 라벨
* 첨부 보상 미리보기
* 만료까지 남은 시간 또는 만료일
* 읽음/안읽음 상태 표시
* 수령 가능 여부 표시

### 하단/보조 영역

* 우편이 없을 경우 빈 상태 문구 표시
* 예: “도착한 우편이 없습니다.”

## 4-3. 상세 팝업 구성

우편 클릭 시 상세 팝업 노출

구성 요소:

* 제목
* 본문
* 첨부 보상 목록
* 만료일
* 개별 수령 버튼
* 닫기 버튼

## 4-4. 버튼 규칙

### 전체 수령

* 수령 가능한 모든 우편의 보상을 일괄 수령
* 공지 우편은 수령 대상에서 제외
* 처리 후 결과 팝업 또는 토스트 출력

예:

* 골드 100,000 획득
* 장비 소환권 20 획득

### 개별 수령

* 선택한 우편의 첨부 보상만 수령

## 4-5. 추천 UI 플로우

`메인 화면 → 우편 버튼 → 우편 리스트 → 우편 선택 → 상세 확인 → 수령`

## 4-6. UX 포인트

* 수령 가능한 우편이 위로 오도록 정렬 우선순위 부여 가능
* 첨부 보상이 있는 우편은 보상 아이콘으로 직관성 강화
* 전체 수령 후 자동으로 리스트 새로고침
* 수령 후 해당 우편 상태를 즉시 “수령 완료”로 변경

## 5. 예외 처리

## 5-1. 만료된 우편 수령 시도

### 상황

유저가 우편 상세를 열어둔 상태에서 우편이 만료됨

### 처리

* 수령 요청 시 서버에서 만료 여부 재검사
* 만료되었으면 수령 불가 반환
* 클라이언트는 “우편 보관 기간이 만료되었습니다.” 팝업 표시
* 우편 상태를 Expired로 갱신

## 5-2. 중복 수령 요청

### 상황

네트워크 지연 또는 버튼 연타로 동일 우편에 대해 중복 요청 발생

### 처리

* 서버에서 `is_claimed = true` 여부 확인
* 이미 수령된 경우 보상 재지급 없이 성공/무효 응답 반환
* 클라이언트는 상태만 최신화

## 5-3. 전체 수령 중 일부 우편 만료

### 상황

전체 수령 처리 중 일부 우편이 이미 만료 상태

### 처리

* 서버는 수령 가능한 우편만 처리
* 응답에 성공 수, 실패 수, 만료 수 포함
* 클라이언트는 결과 메시지 출력

예:

* 5개 우편 중 4개 수령 완료, 1개 만료

## 5-4. 보상 지급 실패

### 상황

DB 반영 실패, 트랜잭션 오류, 계정 데이터 저장 실패

### 처리

* 우편 수령과 재화 지급은 하나의 트랜잭션으로 묶음
* 일부만 지급된 상태를 허용하지 않음
* 실패 시 전체 롤백
* 유저에게는 “보상 수령에 실패했습니다. 다시 시도해 주세요.” 표시

## 5-5. 우편 없음 상태

### 상황

우편함이 비어 있음

### 처리

* 빈 상태 문구 출력
* 전체 수령 버튼 비활성화

## 5-6. 오래된 리스트 캐시 문제

### 상황

클라이언트가 오래된 우편 상태를 표시 중

### 처리

* 우편함 진입 시 서버 최신 데이터 재조회
* 수령/읽기 이후 목록 재동기화
* 로컬 상태는 연출용으로만 사용하고 최종 판단은 서버 기준

## 5-7. 인벤토리 초과 문제

현재 기획상 우편 보상은 **재화/티켓만 지급**하므로 인벤토리 부족 문제는 발생하지 않는다.
향후 직접 장비 지급이 추가될 경우, 해당 시점에 별도 예외 처리를 설계한다.

## 6. 서버/클라이언트 처리 구분

## 6-1. 서버 담당

서버는 우편 시스템의 **권한 주체**다.

### 서버 처리 항목

* 우편 생성
* 대량 발송
* 지급 대상 판별
* 만료 시간 관리
* 우편 목록 조회
* 읽음 상태 저장
* 개별 수령 처리
* 전체 수령 처리
* 재화 지급 처리
* 중복 수령 방지
* 운영툴 연동
* 로그 기록

### 서버 권한 원칙

* 보상 수량 결정은 서버가 한다.
* 클라이언트는 우편 정보를 표시만 하며 보상 수량을 확정하지 않는다.
* 수령 가능 여부, 만료 여부, 중복 수령 여부는 모두 서버에서 판정한다.

## 6-2. 클라이언트 담당

클라이언트는 **표시와 입력 전달**을 담당한다.

### 클라이언트 처리 항목

* 우편 리스트 UI 렌더링
* 레드닷 표시
* 우편 상세 팝업 출력
* 수령 버튼 입력 처리
* 수령 결과 연출
* 빈 상태 표시
* 임시 로컬 상태 반영(낙관적 UI 가능)

## 6-3. 권장 API 예시

### 우편 목록 조회

`GET /mail/list`

응답:

* 우편 기본 목록
* 레드닷 계산용 상태
* 수령 가능 개수

### 우편 상세 조회

`GET /mail/detail/{mail_id}`

### 우편 읽음 처리

`POST /mail/read`

### 개별 수령

`POST /mail/claim`

### 전체 수령

`POST /mail/claim-all`

| 보상 계산  | O      | X     |

### 보상 경제 밸런스 체크

우편은 운영자가 재화를 직접 지급할 수 있는 수단이므로,
아래 항목을 주기적으로 점검해야 한다.

* 일/주간 총 우편 지급량
* 재화별 지급량 추이
* 점검 보상 과다 지급 여부
* 이벤트 보상 과잉 여부
* 특정 계정 오지급 여부

* 우편 수령률
* 전체 수령 사용률
* 평균 미수령 우편 수

* 보상량 상한 검증
* 비정상적으로 큰 지급량 탐지

## 7-5. 추천 운영 정책

* 점검 보상: 점검 종료 후 즉시 우편 발송
* 랭킹 보상: 일일 또는 주간 정산 후 우편 발송
* 시즌패스 미수령 보상: 시즌 종료 시 자동 우편 발송 가능
* 이벤트 보상: 조건 달성 즉시 또는 이벤트 종료 후 일괄 발송

# 간단 요약

이 우편 시스템의 핵심은 아래다.

* **유저 간 우편 없음**
* **운영/보상 전달 전용**
* **재화와 티켓만 첨부**
* **개별 수령 + 전체 수령 지원**
* **만료, 중복 수령, 롤백을 서버에서 엄격 처리**
* **운영 보상 지급 허브로 사용**

원하면 다음으로 이어서 **같은 형식으로 ‘퀘스트’나 ‘오프라인 보상’ 기획서**도 바로 작성해줄게.
