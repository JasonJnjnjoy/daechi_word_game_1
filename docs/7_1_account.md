좋아. 아래는 네 기준에 맞춰 정리한 **방치형 게임 ‘계정’ 시스템 기획서**야.
최대한 **단순 운영형**, **모바일 idle game에 맞는 구조**, **서버 권한 중심**으로 잡았어.

# 계정 시스템 기획서

## 1. 시스템 개요

### 1-1. 목적

계정 시스템은 유저 식별, 인증, 데이터 영속화, 기기 변경 대응, 제재 관리, 고객 지원 대응을 위한 기본 시스템이다.
이 게임은 구조를 단순화하므로 **1계정 = 1게임 프로필**을 기본 원칙으로 한다.

### 1-2. 시스템 역할

* 유저의 로그인/인증 처리
* 계정 고유 ID 발급 및 관리
* 게임 데이터와 계정 연결
* 게스트 계정 및 외부 계정 연동
* 제재 상태 관리
* 탈퇴/복구/문의 대응용 기준 데이터 제공

### 1-3. 설계 방향

* 초반 진입장벽을 낮추기 위해 **게스트 로그인 지원**
* 데이터 유실 방지를 위해 **외부 계정 연동 유도**
* 운영 단순화를 위해 **멀티 캐릭터 미지원**
* 치트 방지 및 데이터 정합성을 위해 **주요 상태는 서버 권한**
* 오프라인 보상, 길드, 랭킹 등은 모두 계정 단위로 귀속

### 1-4. 계정 종류

* 게스트 계정
* Google 계정
* Apple 계정
* (선택) 이메일 계정
  ※ 초기 출시에서는 **게스트 + Google + Apple**만으로도 충분

## 2. 규칙 설명

### 2-1. 계정 생성 규칙

* 최초 실행 시 로그인 선택 화면 진입
* 유저는 아래 중 하나를 선택 가능

  * 게스트로 시작
  * Google 로그인
  * Apple 로그인
* 게스트 계정도 즉시 플레이 가능
* 서버는 계정 생성 시 `account_id`를 발급

### 2-2. 기본 운영 규칙

* 1계정은 1개의 게임 진행 데이터만 가진다
* 닉네임은 계정 생성 후 최초 1회 설정
* 닉네임 변경은 무료 1회 또는 유료 정책 중 택1
  → 단순화를 위해 초기 버전은 **무료 1회 + 이후 다이아 소모** 권장
* 계정 데이터는 서버에 저장
* 로컬 데이터는 자동 로그인 정보/기기 식별 보조 정보만 저장

### 2-3. 게스트 계정 규칙

* 게스트 계정은 앱 삭제, 기기 교체, OS 초기화 시 데이터 유실 가능
* 게스트 계정은 로그인 연동 팝업을 통해 외부 계정 연동 유도
* 게스트 계정은 외부 계정으로 **승격 연결** 가능
* 이미 다른 계정에 연결된 외부 계정에는 중복 연결 불가

### 2-4. 외부 계정 연동 규칙

* Google/Apple 계정은 1개의 게임 계정에만 연결 가능
* 게스트 → 외부 계정 연동은 가능
* 외부 계정 A → 외부 계정 B 추가 연동도 가능
  예: Apple로 시작 후 Google 추가 연결
* 연동 해제는 최소 1개 로그인 수단이 남아 있을 때만 가능

### 2-5. 탈퇴 규칙

* 탈퇴 신청 시 즉시 삭제하지 않고 **유예 기간** 운영 권장
  예: 7일
* 유예 중 로그인 시 탈퇴 취소 가능
* 탈퇴 완료 후 게임 데이터는 복구 불가 정책 적용 가능
* 운영툴에서는 일정 기간 백업 보관 가능

### 2-6. 제재 규칙

* 제재 상태 종류

  * 정상
  * 채팅 제한
  * 길드 기능 제한
  * 랭킹 제외
  * 로그인 정지
  * 영구 정지
* 제재 사유와 시작/종료 시각은 서버 저장
* 제재 여부는 로그인 시점 및 주요 기능 진입 시점에 서버에서 확인

### 2-7. 중복 접속 규칙

* 동일 계정 동시 접속은 기본적으로 1세션만 허용
* 다른 기기에서 로그인 시 기존 세션 무효화
* PvP 실시간 전투가 없으므로 강한 세션 동기화는 불필요
* 계정 도용 방지를 위해 중요 시점에 토큰 재검증

### 2-8. 연령/약관 규칙

* 최초 로그인 시 약관/개인정보 처리방침 동의 필요
* 선택 약관(마케팅 수신)은 분리 저장
* 미동의 시 게임 시작 불가

## 5. 예외 처리

### 5-1. 게스트 계정 유실

* 앱 재설치/기기 변경으로 게스트 데이터가 사라질 수 있음
* 서버에 계정이 남아 있어도 로그인 키를 잃으면 복구 어려움
* 운영 CS에서는 `account_id`, 결제 영수증, 닉네임 등으로 탐색 시도 가능
* 정책상 복구 불가 또는 제한 복구 중 하나를 명확히 해야 함

### 5-2. 외부 로그인 실패

* 플랫폼 인증 실패 시 재시도 유도
* 서버 연동 실패 시 “일시적 오류” 메시지 출력
* 외부 토큰은 유효하지만 서버 계정 생성 실패 시 중복 생성 방지 필요

### 5-3. 이미 다른 계정에 연결된 외부 계정

* 연동 시도 차단
* “이미 다른 게임 계정에 연결된 로그인 수단입니다.” 안내

### 5-4. 닉네임 중복/금칙어

* 생성 불가
* 금칙어는 일반 금칙어 + 운영 지정 금칙어 분리 관리

### 5-5. 중복 접속

* 기존 세션 만료 처리
* 만료된 클라이언트는 재로그인 유도 팝업 표시

### 5-6. 탈퇴 유예 중 재접속

* 로그인 시 탈퇴 예정 상태 안내
* 유저가 원하면 탈퇴 취소 가능

### 5-7. 제재 상태에서 기능 접근

* 로그인 가능하지만 일부 기능만 제한될 수 있음
* 기능 버튼 비활성화 또는 진입 후 안내 팝업 처리

### 5-8. 점검 중 로그인

* 로그인 차단
* 점검 시간 및 보상 안내 팝업 표시

## 6. 서버/클라이언트 처리 구분

## 6-1. 서버 처리

서버 권한 처리 대상:

* 계정 생성/인증
* 외부 로그인 토큰 검증
* 계정 연동/해제
* 닉네임 중복 검사 최종 승인
* 제재 상태 조회
* 탈퇴 신청/취소
* 계정 세션 발급/만료
* 계정 데이터 로드/저장
* 약관 동의 이력 저장

서버 판단 이유:

* 계정 도용 방지
* 중복/악용 방지
* 고객센터/운영 대응 기준 일원화
* 랭킹/길드/오프라인 보상과의 정합성 유지

## 6-2. 클라이언트 처리

클라이언트 담당:

* 로그인 UI 표시
* 외부 플랫폼 SDK 호출
* 서버에서 받은 계정 상태 화면 반영
* 연동 유도 팝업 노출
* 계정 정보 표시
* 오류 메시지/재시도 UX 제공
* 토큰 로컬 저장

## 6-3. 클라이언트 저장 최소화 원칙

클라이언트에는 아래 정도만 저장

* access token / refresh token
* 마지막 로그인 수단
* 간단한 기기 식별 정보
* 로그인 상태 캐시

게임 핵심 진행 데이터는 저장하지 않음
→ 변조 방지 목적

* 비정상 로그인 시도 로그

# 추가 정책 제안

## A. 이 게임에 맞는 추천 운영안

이 프로젝트는 단순형 방치 게임이므로 계정도 복잡하게 갈 필요가 없다.
초기 버전 추천안은 아래와 같다.

* 로그인 방식: 게스트 / Google / Apple
* 1계정 1프로필
* 서버 저장 필수
* 게스트 연동 강하게 유도
* 닉네임 중복 허용 안 함
* 동시 접속 1기기만 허용
* 탈퇴 유예 7일
* 제재는 로그인 제한 / 채팅 제한 / 랭킹 제외만 우선 구현

## B. MVP 우선순위

### 필수

* 게스트 로그인
* Google/Apple 로그인
* 계정 생성
* 닉네임 생성
* 자동 로그인
* 제재 체크
* 계정 연동
* 탈퇴 신청

### 후순위

* 닉네임 변경권
* 연동 해제
* 로그인 기록 상세 조회
* 비정상 접속 알림
