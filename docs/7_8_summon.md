아래는 **장비 소환 + 스킬 소환을 포함한 통합 소환 시스템** 기준의 기획서입니다.
가급적 단순하게 유지하기 위해 **상시 배너 2종만 운영**하고, **조각/파편 재화는 추가하지 않는 구조**로 잡았습니다.

# 소환 시스템 기획서

## 전제

* 소환 시스템은 **장비 소환**, **스킬 소환** 2종으로 구성한다.
* 소환 결과물은 **장비** 또는 **스킬**만 지급한다.
* 기본 소모 재화는

  * 장비 소환: **장비 소환권**
  * 스킬 소환: **스킬 소환권**
* 다이아 직접 소환은 초기 버전에서는 **선택 기능**으로 두고, 운영 플래그로 ON/OFF 가능하게 설계한다.
* 스킬 장착/프리셋은 **#21 스킬 시스템**에서 처리하며, 본 문서는 **#15 스킬 획득/강화용 소환**만 다룬다.

---

## 1. 시스템 개요

### 1-1. 목적

소환 시스템의 목적은 다음과 같다.

* 유저의 반복 플레이 보상을 **즉시 성장 재화**로 전환하는 핵심 창구 제공
* 지역 진행이 막혔을 때 **성장 우회 수단** 제공
* 시즌패스 / 이벤트 / 길드 / 상점에서 지급한 티켓의 **소비처** 제공
* 장기적으로는 **소환 레벨**과 **확정 보정(Pity)** 를 통해 지속 플레이 동기 제공

### 1-2. 핵심 구성

* 상시 배너 2종

  * 장비 소환
  * 스킬 소환
* 소환 방식 2종

  * 1회 소환
  * 10회 소환
* 성장 장치 2종

  * 배너별 **소환 레벨**
  * 배너별 **확정 게이지(Pity)**

### 1-3. 해금 시점

* **튜토리얼 종료 후 즉시 해금**
* 초반 성장 루프와 직결되는 핵심 기능이므로 별도 늦은 해금은 두지 않는다.

### 1-4. 설계 원칙

* 복잡한 한정 배너 / 픽업 배너 / 스텝업 배너는 **초기 버전에서 제외**
* 장비와 스킬은 모두 **템플릿 기반 획득**으로 처리하여 데이터와 운영을 단순화
* 소환 결과 계산은 반드시 **서버 권한**으로 처리
* 소환 결과는 즉시 지급하며, 별도 수동 수령 과정은 두지 않음

### 1-5. 제외 범위

다음 요소는 본 시스템 범위에서 제외한다.

* 펫 / 부적 / 마정석 / 연구소 관련 소환
* 한정 픽업 배너
* 광고 무료 소환
* 파편/조각 합성 시스템
* PvP 전용 보상 가챠

## 2. 규칙 설명

### 2-1. 기본 규칙

* 소환 탭은 **장비 소환 / 스킬 소환** 2개만 존재한다.
* 각 탭은 서로 독립된 상태값을 가진다.

  * 소환 레벨 독립
  * 확정 게이지 독립
  * 이력 독립
* 소환은 항상 **현재 열려 있는 상시 배너**에서만 가능하다.

### 2-2. 소모 재화 규칙

#### 기본안

* 장비 소환 1회: 장비 소환권 1장
* 장비 소환 10회: 장비 소환권 10장
* 스킬 소환 1회: 스킬 소환권 1장
* 스킬 소환 10회: 스킬 소환권 10장

#### 선택안: 다이아 직접 소환

* 운영 플래그 `allow_diamond_summon = true` 인 경우에만 허용
* 티켓이 없을 때 다이아로 직접 소환 가능
* 혼합 결제는 허용하지 않음

  * 예: 티켓 7장 + 다이아 3회분 조합 사용 불가
* 소모량은 데이터 테이블로 관리

### 2-3. 소환 결과 산출 규칙

#### 장비 소환

장비 소환은 아래 순서로 처리한다.

1. 등급 결정
2. 부위 결정

   * 무기 / 방어구 / 반지 / 목걸이
3. 해당 등급 + 해당 부위의 풀에서 장비 1개 추첨
4. 결과를 계정에 즉시 지급

#### 장비 부위 확률

* 기본값: **4부위 균등 25%**
* 필요 시 데이터 테이블에서 가중치 조정 가능

#### 스킬 소환

스킬 소환은 아래 순서로 처리한다.

1. 등급 결정
2. 해당 등급의 스킬 풀에서 스킬 1개 추첨
3. 결과를 계정에 즉시 지급

### 2-4. 등급 구조

초기 버전은 단순화를 위해 공통 등급 구조를 사용한다.

* 일반
* 희귀
* 영웅
* 전설

실제 확률은 배너별로 별도 관리 가능하게 한다.

### 2-5. 소환 레벨 규칙

각 배너는 별도의 소환 레벨을 가진다.

* 장비 소환 레벨
* 스킬 소환 레벨

#### 동작 방식

* 소환 1회당 해당 배너에 **소환 경험치 1** 획득
* 누적 경험치가 기준치에 도달하면 소환 레벨 상승
* 레벨이 오르면 다음 소환부터 **상위 등급 확률 증가**
* 소환 레벨 보상은 별도 지급하지 않고, **확률 상승 자체를 보상**으로 사용

#### 예시 레벨 구조

* Lv1 → Lv2 : 20회
* Lv2 → Lv3 : 50회
* Lv3 → Lv4 : 100회
* Lv4 → Lv5 : 200회

실제 수치는 `SummonLevelMaster`에서 조정한다.

#### 예시 확률 테이블

아래는 예시이며 실제 서비스 수치는 운영 조정 가능하다.

| 소환 레벨 구간 |  일반 |  희귀 |   영웅 |   전설 |
| -------- | --: | --: | ---: | ---: |
| Lv1~2    | 70% | 25% | 4.5% | 0.5% |
| Lv3~4    | 58% | 31% | 9.5% | 1.5% |
| Lv5~6    | 45% | 38% |  14% |   3% |
| Lv7~8    | 32% | 45% |  18% |   5% |
| Lv9~10   | 20% | 52% |  22% |   6% |

### 2-6. 확정 보정(Pity) 규칙

유저의 체감 불쾌감을 줄이기 위해 확정 보정을 둔다.

#### 기본 규칙

* **10회 소환 시 최소 1개 희귀 이상 보장**
* **영웅 이상 미획득 49회 연속 시 다음 1회 영웅 이상 확정**
* **전설 미획득 199회 연속 시 다음 1회 전설 확정**

#### 카운터 규칙

* 장비 소환과 스킬 소환의 확정 카운터는 별도 관리
* 전설 획득 시

  * 전설 카운터 초기화
  * 영웅 카운터도 초기화
* 영웅 획득 시

  * 영웅 카운터 초기화
  * 전설 카운터는 유지

#### 우선순위

소환 1회 처리 시 우선순위는 다음과 같다.

1. 전설 확정 체크
2. 영웅 확정 체크
3. 일반 확률 추첨
4. 10회 소환의 희귀 보정 체크

즉, 상위 확정이 먼저 적용된다.

### 2-7. 중복 획득 처리

#### 장비 중복

* 장비는 **랜덤 옵션 없는 템플릿형 장비**로 설계
* 동일 장비를 또 획득하면 **보유 수량 +1**
* 이후 장비 강화/승급 시스템에서 소모 가능

#### 스킬 중복

* 미보유 스킬이면 **신규 해금**
* 이미 보유 중이면 **강화 재료용 복사본 수량 +1**
* 이미 최대 강화 상태인 스킬을 추가 획득하면 **골드로 자동 환산**

  * 별도 신규 재화는 추가하지 않음

### 2-8. 지급 규칙

* 소환 결과물은 즉시 계정에 반영한다.
* 우편 지급은 사용하지 않는다.
* 10회 소환도 **한 번의 원자적 트랜잭션**으로 처리한다.

  * 전부 성공 또는 전부 실패
* 소환 결과는 최근 이력에 저장한다.

### 2-9. 무료 소환 정책

* 초기 버전은 **기본 무료 소환 없음**
* 무료 지급은 시즌패스 / 이벤트 / 점검 보상 / 길드 보상으로 대체
* 이유:

  * 경제 통제 단순화
  * 접속 강제 압박 감소
  * 무료 소환 루틴 없이도 보상 구조 설계 가능

소환 시스템 데이터는 **마스터 데이터**, **유저 상태 데이터**, **트랜잭션 데이터**로 나눈다.

* `request_id`
* `user_id`
* `banner_id`
* `summon_type`
* `pull_count`
* `cost_type`

  * ticket / diamond
* `cost_amount`
* `before_currency_json`
* `after_currency_json`
* `before_state_json`
* `after_state_json`
* `pity_trigger_json`

클라이언트 표시용 이력 뷰

* 최근 N건 결과 표시용
* 전설 획득 시점, 소환 종류, 횟수, 주요 결과를 빠르게 조회 가능하도록 별도 인덱싱

### 3-4. 데이터 구조 설계 원칙

* 확률표와 풀은 반드시 분리한다.
* 배너와 풀은 데이터로 연결한다.
* 등급 확률, 확정 규칙, 레벨 기준은 모두 **라이브 수정 가능한 마스터 데이터**로 관리한다.
* 소환 결과 원복/분쟁 대응을 위해 `request_id` 기반 재조회가 가능해야 한다.

## 4. UI 구성

### 4-1. 화면 구성 목표

* 한 화면에서 정보 이해와 소환 실행이 모두 가능해야 한다.
* 유저가 헷갈리지 않도록 배너 수를 최소화한다.
* 확률, 확정 게이지, 소환 레벨을 항상 노출해 신뢰감을 높인다.

### 4-2. 메인 화면 구성

```text
[상단 재화 바]
골드 / 다이아 / 장비 소환권 / 스킬 소환권

[탭]
장비 소환 | 스킬 소환

[중앙 배너 영역]
배너 일러스트 또는 대표 보상 프리뷰
소환 레벨 n
소환 경험치 바
전설 확정까지 xx회 / 영웅 확정까지 xx회
확률 정보 버튼

[하단 실행 버튼]
1회 소환
10회 소환

[보조 버튼]
확률 보기
획득 이력
연출 스킵
```

### 4-3. 필수 UI 요소

#### 공통

* 현재 보유 티켓 수
* 다이아 수
* 현재 소환 레벨
* 다음 레벨까지 진행도
* 영웅/전설 확정까지 남은 횟수
* 1회 / 10회 버튼
* 확률 상세 팝업
* 소환 결과 팝업

#### 장비 소환 전용

* 부위 대표 아이콘 표시
* 현재 풀에 포함된 대표 장비 미리보기

#### 스킬 소환 전용

* 대표 스킬 아이콘 표시
* 신규 획득 가능 스킬/강화 대상 스킬 미리보기

### 4-4. 결과 팝업

결과 팝업에는 다음이 포함된다.

* 획득 아이템/스킬 1개 또는 10개 목록
* 신규 획득 여부 표시
* 중복 획득 여부 표시
* 영웅 이상 획득 시 강조 연출
* `다시 1회`, `다시 10회`, `닫기` 버튼

### 4-5. UX 원칙

* 10회 소환 결과는 한 번에 보여주되, 전설이 있으면 해당 카드만 확대 연출
* 티켓이 없을 경우 버튼 비활성화 또는 다이아 대체 확인 팝업 표시
* 초기 버전에서는 자동 연속 소환 기능은 제외하여 실수 소비를 줄인다.

## 5. 예외 처리

### 5-1. 기능 잠금 상태

* 아직 소환 기능이 해금되지 않은 경우:

  * 버튼 비활성화
  * “튜토리얼 완료 후 이용 가능합니다” 등 문구 표시

### 5-2. 재화 부족

* 티켓 부족 시:

  * 기본안: 소환 불가 팝업 표시
  * 선택안: 다이아 대체 사용 여부 확인 팝업 표시
* 다이아도 부족하면 소환 불가

### 5-3. 배너 비활성 상태

* 점검 중 또는 운영 플래그 OFF 시:

  * 진입 불가
  * “현재 이용할 수 없습니다” 메시지 표시

### 5-4. 중복 요청 / 네트워크 재시도

* 클라이언트는 요청마다 `request_id`를 생성
* 동일 `request_id` 재전송 시 서버는 **기존 결과를 재반환**
* 중복 차감이 발생하지 않도록 한다.

### 5-5. 트랜잭션 중 실패

* 비용 차감 후 결과 지급 전에 에러가 발생하면 전체 롤백
* 10회 소환은 부분 성공을 허용하지 않음

### 5-6. 클라이언트와 서버 데이터 불일치

* 클라이언트가 본 확률/풀 정보와 서버 최신 데이터가 다른 경우:

  * 서버 데이터 기준으로 처리
  * 응답 시 클라이언트에 최신 마스터 버전 전달
  * 필요 시 UI 새로고침 유도

### 5-7. 최대 강화 스킬 중복 획득

* 이미 최대 강화 상태인 스킬을 획득하면:

  * 즉시 골드 환산
  * 결과 팝업에 “최대 강화 스킬 중복 → 골드 전환” 문구 표기

### 5-8. 인벤토리 관련

* 장비는 템플릿 스택형이므로 **인벤토리 가득 참 예외 없음**
* 장비 개별 인스턴스 저장 구조를 사용하지 않아 운영 복잡도 최소화

### 5-9. 접속 종료/재접속

* 소환 도중 앱이 종료되어도 서버 트랜잭션이 완료되었다면 결과는 계정에 반영됨
* 재접속 시 마지막 `tx_id` 또는 `request_id`로 결과 복구 조회 가능해야 함

### 6-1. 서버 책임

서버는 반드시 다음을 담당한다.

* 비용 차감
* 확률/확정 계산
* 지급 대상 산출
* 유저 상태 변경 저장
* 이력 기록
* 중복 요청 방지
* 이상 요청 탐지

### 6-2. 클라이언트 책임

클라이언트는 다음만 담당한다.

* 서버 응답 결과 시각화
* 최신 마스터 데이터 표시

### 6-3. 처리 원칙

* 소환 결과는 항상 서버 응답을 기준으로 렌더링한다.
* 10회 소환은 서버에서 10개 결과를 모두 확정한 뒤 한 번에 반환한다.

## 7. 로그 및 운영 포인트

### 7-1. 필수 로그

#### 소환 요청 로그

* 누가
* 어떤 배너에서
* 몇 회를
* 어떤 재화로 실행했는지

#### 소환 결과 로그

* 각 결과의 등급
* reward_id
* 확정 보정 발동 여부
* 신규 획득 여부
* 최대 강화 중복 환산 여부

#### 상태 변화 로그

* 소환 전/후 티켓 수
* 소환 전/후 다이아 수
* 소환 전/후 소환 레벨
* 확정 카운터 변화

#### 예외 로그

* 중복 요청
* 비정상 반복 호출
* 재화 부족 상태에서의 반복 요청
* 마스터 데이터 누락
* 트랜잭션 실패/롤백

### 7-2. 운영 모니터링 지표

* 일간 소환 참여 유저 수
* 유저 1인당 평균 소환 횟수
* 장비 소환 / 스킬 소환 비율
* 티켓 공급량 대비 소비량
* 영웅/전설 실제 획득 비율
* 영웅/전설 평균 획득 회수
* 확정 보정 발동 비율
* 최대 스킬 중복 골드 환산량

### 7-3. 핵심 운영 포인트

#### 1) 티켓 공급처 밸런스

소환 재미는 티켓 공급량에 크게 좌우되므로 아래 공급처 비율을 조정해야 한다.

* 오프라인 보상
* 던전 보상
* 시즌패스
* 이벤트
* 길드 상점/길드 보상
* 상점 패키지

#### 2) 장비 소환 vs 스킬 소환 비중

둘 중 하나만 과도하게 공급되면 성장 체감이 쏠리므로,

* 장비 성장
* 스킬 성장
  의 시간당 효율을 지속적으로 비교해야 한다.

#### 3) 확정 게이지 체감

확정 천장이 너무 멀면 불쾌감이 커지고,
너무 짧으면 희소성이 무너진다.
초기 서비스에서는 전설보다 **영웅 체감**을 먼저 안정화하는 것이 중요하다.

#### 4) 소환 레벨 성장 속도

소환 레벨이 너무 빨리 오르면 장기 동기가 약해지고,
너무 느리면 성장이 안 느껴진다.
초기 3일 내에 최소 1~2회 레벨업 체감이 있어야 한다.

#### 5) 운영 단순화 방침

초기 라이브 운영은 아래처럼 단순하게 가는 것이 좋다.

* 상시 배너 2종만 운영
* 확률 변경 최소화
* 픽업/한정 배너 미도입
* 이벤트는 **티켓 지급량 조정 중심**
* 확률 테이블보다 **재화 공급량**으로 체감 조절

### 7-4. 고객 대응 포인트

분쟁 대응을 위해 다음이 가능해야 한다.

* 특정 유저의 소환 이력 조회
* request_id 기준 결과 재확인
* 전설 획득 여부 추적
* 비정상 차감 여부 확인
* 보상 누락 CS 대응용 재지급 판단 근거 확보
