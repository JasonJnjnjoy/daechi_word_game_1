# 일일 출석 보상 시스템 기획서

본 문서는 방치형 게임의 **일일 출석 보상 시스템**을 정의한다.
이 시스템은 유저의 일일 접속 동기를 만들고, 코어 루프에 필요한 재화를 안정적으로 공급하기 위한 장치다.

이 프로젝트는 복잡도를 줄이기 위해 **월간 달력형 출석이 아닌, 7일 누적 순환형 출석**을 사용한다.
또한 UX 단순화를 위해 **수동 수령이 아닌 자동 수령 방식**을 기본으로 한다.

---

## 1. 시스템 개요

## 1-1. 시스템 목적

일일 출석 보상 시스템의 목적은 다음과 같다.

* 하루 1회 접속 동기 제공
* 골드, 소환권, 던전 티켓, 다이아 등 핵심 재화의 안정적 공급
* 무과금/저과금 유저의 기본 성장 리듬 유지
* 시즌패스, 이벤트, 던전, 소환 시스템으로 자연스럽게 연결

본 게임의 코어 루프와 연결하면 다음과 같은 역할을 가진다.

* 출석 보상으로 재화 획득
* 획득한 재화로 성장/소환/던전 진입
* 더 높은 지역 진입 및 반복 플레이 유도

---

## 1-2. 기본 설계 방향

### 기본 정책

* **7일 누적 순환형**
* **연속 출석 강제 없음**
* **하루 1회 자동 수령**
* **계정 단위 관리**
* **보상은 현재 사용 중인 6종 재화만 지급**

### 지원 보상 종류

* 골드
* 다이아
* 장비 소환권
* 스킬 소환권
* 던전 티켓
* 길드 코인

---

## 1-3. 핵심 특징

1. 유저가 하루라도 접속하면 해당 일자의 출석 보상을 받을 수 있다.
2. 하루를 놓쳐도 출석 진행도는 초기화되지 않는다.
3. 7일차 보상을 수령하면 다음 출석은 다시 1일차부터 시작된다.
4. 별도 클릭 없이 **로그인 후 메인 로비 진입 시 자동 지급**된다.
5. 보상 구조는 운영툴에서 버전 단위로 관리 가능하다.

---

## 2. 규칙 설명

## 2-1. 출석 인정 규칙

### 출석 인정 기준

* 서버 기준 일일 리셋 이후
* 해당 계정이 처음으로 **메인 로비 진입**에 성공하면
* 서버가 출석 가능 여부를 판정하고 보상을 자동 지급한다.

즉, 단순 앱 실행만으로는 확정되지 않고, **서버와 정상 통신이 완료된 로비 진입 시점**에 출석이 인정된다.

---

## 2-2. 일일 수령 규칙

1. 출석 보상은 **하루 1회만 수령 가능**하다.
2. 같은 날 여러 번 접속해도 추가 지급되지 않는다.
3. 일일 리셋 시각은 **서버 기준 매일 00:00**로 한다.
4. 클라이언트 로컬 시간은 신뢰하지 않는다.
5. 출석 보상은 자동 지급되며, 별도 “받기” 버튼은 두지 않는다.

---

## 2-3. 누적 진행 규칙

### 누적형 출석

* 출석은 **연속형이 아니라 누적형**이다.
* 하루를 놓쳐도 출석 일수는 유지된다.
* 다음 접속 시 이전 진행도 다음 일차 보상부터 이어서 수령한다.

예시:

* 월요일 1일차 수령
* 화요일 미접속
* 수요일 접속 시 2일차 수령

### 순환 구조

* 7일차 수령 완료 시 1사이클 종료
* 다음 출석부터 새로운 사이클의 1일차 시작

---

## 2-4. 신규 유저 규칙

* 신규 계정은 첫 로그인 후 첫 로비 진입 시 **1일차 보상**을 자동 수령한다.
* 별도의 해금 조건은 두지 않는다.
* 길드 미가입 상태여도 길드 코인을 지급받을 수는 있으나, 기본 보상표에는 길드 코인을 초반 보상으로 넣지 않는 것을 권장한다.

---

## 2-5. 온라인 상태 리셋 대응 규칙

유저가 리셋 시각을 넘겨 계속 접속 중일 경우에도 아래 조건에서 출석 동기화가 가능해야 한다.

* 메인 로비 재진입 시
* 앱 포그라운드 복귀 시
* 서버 동기화 재요청 시

즉, “로그인”만이 아니라 **리셋 이후 최초의 로비 동기화 시점**에 자동 수령 가능해야 한다.

---

## 2-6. 기본 보상 예시

아래는 권장 예시이며, 실제 수치는 밸런스 테이블에서 조정한다.

| 일차  | 보상 예시                         | 설계 의도       |
| --- | ----------------------------- | ----------- |
| 1일차 | 골드 10,000                     | 가장 기본 성장 지원 |
| 2일차 | 장비 소환권 3                      | 장비 성장 유도    |
| 3일차 | 스킬 소환권 3                      | 스킬 성장 유도    |
| 4일차 | 골드 20,000                     | 중반 성장 보강    |
| 5일차 | 던전 티켓 2                       | 던전 플레이 유도   |
| 6일차 | 다이아 100                       | 프리미엄 보상 체감  |
| 7일차 | 장비 소환권 5 + 스킬 소환권 5 + 다이아 200 | 주간 보상 피크    |

### 보상 설계 원칙

* 초반은 골드와 소환권 중심
* 중후반은 티켓/다이아 중심
* 7일차는 확실한 보상 체감을 제공
* 장기적으로는 소환/던전/상점 진입과 연결되도록 설계

---

## 2-7. 보상 버전 적용 규칙

운영 중 보상표가 변경될 수 있으므로 버전 개념을 둔다.

### 기본 정책

* 유저는 **현재 진행 중인 7일 사이클 동안 동일한 보상 버전**을 유지한다.
* 7일차 수령 후 새 사이클이 시작될 때 최신 활성 버전을 적용한다.

이 방식은 운영 중 보상 변경이 있어도, 유저가 사이클 중간에 다른 보상표로 바뀌는 문제를 막는다.

## 3. 데이터 구조

일일 출석 보상 시스템은 **정책 마스터**, **보상 마스터**, **유저 진행 데이터**, **수령 로그**로 구성한다.

---

## 3-1. 출석 정책 마스터

### `attendance_policy_master`

| 필드명            | 타입                | 설명           |
| -------------- | ----------------- | ------------ |
| policy_version | int PK            | 출석 보상 정책 버전  |
| policy_name    | varchar(100)      | 운영용 정책 이름    |
| cycle_length   | int               | 순환 길이, 기본값 7 |
| reset_hour     | tinyint           | 일일 리셋 시각     |
| auto_claim     | bool              | 자동 수령 여부     |
| start_at       | datetime          | 정책 시작 시각     |
| end_at         | datetime nullable | 정책 종료 시각     |
| is_active      | bool              | 현재 활성 여부     |
| created_at     | datetime          | 생성 시각        |
| updated_at     | datetime          | 수정 시각        |

---

## 3-2. 출석 보상 마스터

### `attendance_reward_master`

| 필드명                  | 타입          | 설명                                                                         |
| -------------------- | ----------- | -------------------------------------------------------------------------- |
| attendance_reward_id | bigint PK   | 보상 ID                                                                      |
| policy_version       | int FK      | 정책 버전                                                                      |
| day_no               | tinyint     | 1~7일차                                                                      |
| reward_seq           | tinyint     | 다중 보상 순서                                                                   |
| reward_type          | varchar(32) | gold / diamond / equip_ticket / skill_ticket / dungeon_ticket / guild_coin |
| reward_amount        | bigint      | 지급 수량                                                                      |

하루에 여러 재화를 동시에 줄 수 있도록 1:N 구조로 설계한다.

---

## 3-3. 유저 출석 진행 데이터

### `user_attendance`

| 필드명                   | 타입                | 설명                     |
| --------------------- | ----------------- | ---------------------- |
| user_id               | bigint PK         | 유저 ID                  |
| active_policy_version | int nullable      | 현재 사이클에 묶인 정책 버전       |
| next_reward_day_no    | tinyint           | 다음 수령 예정 일차 (1~7)      |
| last_claim_date_key   | int nullable      | 마지막 수령 일자 키 (YYYYMMDD) |
| last_claim_at         | datetime nullable | 마지막 수령 시각              |
| total_claim_count     | int               | 누적 수령 횟수               |
| cycle_count           | int               | 완료한 7일 사이클 수           |
| updated_at            | datetime          | 최종 갱신 시각               |

### 동작 예시

* 신규 계정: `next_reward_day_no = 1`
* 1일차 수령 후: `next_reward_day_no = 2`
* 7일차 수령 후: `next_reward_day_no = 1`, `cycle_count +1`, `active_policy_version = null`

---

## 3-4. 수령 로그

## 4. UI 구성

## 4-1. 메인 로비 진입 시 자동 노출

리셋 이후 처음으로 로비에 진입해 보상이 지급되면, 출석 결과 팝업을 자동 노출한다.

### 팝업 표시 항목

* “오늘의 출석 보상을 받았습니다”
* 오늘 수령한 보상 목록
* 현재 출석 진행도 (예: 3/7)
* 다음 보상 미리보기
* 확인 버튼

---

## 4-2. 출석 전용 화면

메인 로비 또는 이벤트/보상 메뉴 내에 **출석** 버튼을 둔다.

### 구성 요소

* 7칸 출석 보드
* 각 일차별 보상 아이콘/수량
* 현재 진행 상태 표시
* 오늘은 이미 수령 완료 문구
* 다음 리셋까지 남은 시간 표시

### 상태 표현

* 이미 수령한 날: 완료 표시
* 오늘 수령한 날: 강조 표시
* 다음 수령 예정 날: 테두리 강조
* 이후 일차: 잠금 또는 흐림 표시

---

## 4-3. 메인 화면 표시

메인 로비에서는 과도한 UI를 피하기 위해 아래 정도만 표시한다.

* 출석 버튼
* 현재 진행도 예: `출석 4/7`
* 오늘 수령 완료 시 빨간 점 없음
* 아직 오늘 출석 동기화가 안 된 경우만 진입 시 자동 팝업 발생

자동 수령 방식이므로 별도의 “받기 가능” 레드닷 의존도는 낮다.

---

## 4-4. 보상 연출

출석 보상 수령 시 보상 팝업은 기존 공통 보상 UI를 재사용한다.

### 연출 원칙

* 같은 재화는 합산 표시
* 희소 재화(다이아, 소환권, 티켓)는 강조
* 연출은 짧고 즉시 스킵 가능
* 보상 수령 후 상단 재화 UI 즉시 갱신

---

## 4-5. 오프라인/재접속 UX

네트워크 문제로 팝업 표시가 누락되더라도, 서버 기준으로 보상이 이미 지급되었다면:

* 재접속 시 현재 재화 수량 반영
* 출석 화면에서 오늘 수령 완료 상태 표시
* 중복 팝업은 띄우지 않음

---

## 5. 예외 처리

| 상황               | 처리 방식                         |
| ---------------- | ----------------------------- |
| 같은 날 여러 번 로그인    | 1회만 지급                        |
| 멀티 디바이스 동시 접속    | 먼저 성공한 1회만 지급                 |
| 로비 진입 중 네트워크 끊김  | 서버 지급 성공 여부 기준으로 재동기화         |
| 출석 보상 지급 중 서버 오류 | 전체 롤백, 수령 처리 금지               |
| 출석 지급 후 로그 저장 실패 | 전체 트랜잭션 롤백                    |
| 클라이언트 시간 변경      | 무시, 서버 시간 기준 판정               |
| 보상 버전이 중간에 변경됨   | 현재 사이클은 기존 버전 유지              |
| 리셋 직전/직후 중복 요청   | `request_id` 및 일자 키 기준 1회만 인정 |
| 던전 티켓 등 보상 상한 초과 | 재화 정책에 따라 우편 전환 또는 지급 제한      |
| 점검 중 로그인         | 출석 처리 없음                      |
| 제재 계정            | 정책에 따라 출석 차단 가능               |
| 서버 롤백 발생         | 수령 로그 기준으로 복구 재처리             |

### 추가 원칙

* 출석 지급과 진행도 증가, 로그 저장은 반드시 원자 처리한다.
* “보상은 지급됐는데 진행도는 안 오른 상태” 또는 그 반대 상태는 허용하지 않는다.

## 6. 서버/클라이언트 처리 구분

| 항목                | 서버     | 클라이언트 |
| ----------------- | ------ | ----- |
| 오늘 수령 가능 여부 판정    | O      | X     |
| 일일 리셋 기준 판정       | O      | X     |
| 적용 정책 버전 결정       | O      | X     |
| 보상 지급             | O      | X     |
| 출석 진행도 저장         | O      | X     |
| 중복 수령 방지          | O      | X     |
| 보상 팝업 표시          | X      | O     |
| 출석 보드 렌더링         | X      | O     |
| 진행도/다음 보상 미리보기 표시 | X      | O     |
| 리셋 타이머 UI         | 기준값 제공 | O     |

### 처리 원칙

* 서버는 **수령 자격 판단과 지급 확정**
* 클라이언트는 **상태 표시와 연출**
* 최종 진실값은 항상 서버 데이터 기준

## 7. 로그 및 운영 포인트

## 7-1. 필수 로그

### 반드시 남겨야 할 로그

* 누가 수령했는지
* 몇 일차를 수령했는지
* 어떤 보상이 지급됐는지

## 7-2. 핵심 운영 지표

1. 일일 출석 수령률

   * DAU 대비 출석 자동 수령 성공 비율

2. 7일 사이클 완주율

   * 1일차 진입 유저 중 7일차까지 도달한 비율

3. 일차별 이탈률

   * 1→2일차, 2→3일차, 6→7일차 전환율

4. 보상 재화별 일일 공급량

   * 출석을 통해 풀린 골드/다이아/티켓 총량

5. 자동 수령 실패율

   * 네트워크/트랜잭션 오류 포함

6. 정책 버전별 성과 비교

   * 보상 버전 변경 전후 접속률/완주율 비교

## 7-3. 운영 포인트

### 운영자가 자주 조정하는 항목

* 7일 보상표
* 다이아/티켓 지급량
* 7일차 보상 강도
* 정책 활성 기간
* 특정 버전 전환 시점

### 추천 운영 방향

* 1~3일차는 가볍고 자주 필요한 보상
* 4~6일차는 플레이 유도형 보상
* 7일차는 확실한 체감 보상
* 다이아는 너무 초반에 과하게 풀지 않음
* 장비/스킬 소환권은 성장 정체 구간 해소용으로 활용

---

## 7-4. 운영상 주의 사항

1. 출석 보상이 너무 약하면 유저가 시스템을 인식하지 못한다.
2. 너무 강하면 출석만으로 성장 곡선이 무너진다.
3. 7일차 보상이 약하면 주간 완주 동기가 사라진다.
4. 연속 출석 리셋 스트레스를 제거한 대신, 7일차 보상 체감은 분명해야 한다.
5. 자동 수령 시스템은 편하지만, 유저가 “무엇을 받았는지” 체감하지 못할 수 있으므로 팝업과 보상 연출이 중요하다.
